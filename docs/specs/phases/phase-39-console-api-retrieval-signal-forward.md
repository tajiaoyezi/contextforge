> **诚实校正（ADR-013，本 phase 关键设计裁断）**：历史上 `[SPEC-DEFER:phase-future.console-api-rerank-forward]`（task-21.2 / task-21.3 / README 记录）设想 console-api 以 `?rerank=true` **per-request 参数**转发 rerank（镜像 `?semantic` / `?hybrid`）。但 **Phase 38（ADR-043 D3）已确立 reranker 为服务端 env 驱动（`CONTEXTFORGE_RERANKER_PROVIDER`）、明确不是 per-request 参数**（ADR-004 local-first + ADR-043 D3：reranker 为服务端 opt-in、不随每查询参数切换）。故「`?rerank=true` per-request 转发」与既定设计**冲突**——若实现它会让对外请求覆盖服务端 reranker 选型，破坏 env 驱动模型。本 phase 据实裁断：**兑现** `console-api-hybrid-forward`（hybrid 确是 per-request 算法路由 flag，与 `?semantic` 同类）；**重界定** `console-api-rerank-forward`——rerank **不**暴露为 per-request 参数（保持 env 驱动，ADR-043 D3），改为使 rerank **provenance（`reason` 标记）在对外 REST 响应可见**（链路已通、缺端到端断言）。`?rerank=true` per-request 控制项据实记为**被 Phase 38 env 驱动决策取代（superseded）**、不实现——以 ADR-044 + ADR-043 add-only Phase-39 Amendment 据实记录，不伪装为「实现了 per-request rerank 转发」（ADR-013 据实分级，不夸大、不矛盾实现）。

## 1. 阶段目标

v0.31.0 ship 后，ContextForge 的 hybrid 融合自 Phase 21（ADR-025）已在 core 层**全实现**——`server.rs` hybrid 路径（`req.hybrid` → `search_hybrid` RRF 融合 → `retrieval_method="hybrid"` + `hybrid_score`，:328-376）+ ## 2. 业务价值

兑现 ADR-025 `[SPEC-DEFER:phase-future.console-api-hybrid-forward]` 并据实重界定 ADR-043 follow-up 的 `[SPEC-DEFER:phase-future.console-api-rerank-forward]`——把「hybrid 融合 + reranker 已在 core / 数据面落地、但对外 REST 不可达 / 不可见」推进到「对外 console-api 首次能请求 hybrid 并看到融合分与 rerank provenance」：

- **hybrid 信号对外不可达**：Phase 21（ADR-025）在 core 层落地 hybrid RRF 融合（`server.rs` hybrid 路径 + `req.hybrid` + `hybrid_score`），Phase 20（ADR-024）贯通了 `?semantic` 对外转发，但 console-api `?hybrid=true` 转发一路诚实延后 `[SPEC-DEFER:phase-future.console-api-hybrid-forward]`——console_data_plane.proto `SearchRequest` 无 `hybrid` 字段、console 数据面无 hybrid dispatch、Go 不转发。本 phase 经 proto add-only + 数据面 dispatch + Go 转发贯通，使「对外 REST 用户能请求 hybrid 融合检索」从延后变为事实。
- **rerank provenance 对外不可见 + per-request 设想与既定设计冲突**：Phase 38（ADR-043 D3）把 reranker 在生产数据面 opt-in 接线（env 驱动），但历史 `[SPEC-DEFER:phase-future.console-api-rerank-forward]` 设想的 `?rerank=true` per-request 转发与 env 驱动模型**冲突**（per-request 覆盖服务端选型）。本 phase 据实重界定：reranker 保持 env 驱动（ADR-043 D3 不变），把 rerank `reason` provenance 在对外 REST 的**可见性**端到端断言（链路已通、缺断言），`?rerank=true` per-request 据实记为被取代、不实现——这是 ADR-013 据实分级（不矛盾实现、不夸大）。
- **`hybrid_score` provenance 字段 parity 缺口**：console_data_plane `SearchResultItem` 有 `vector_score=16`（task-32.3）却无 `hybrid_score`，而 `v1 RetrievalResult` 有 `hybrid_score=15`——对外 hybrid 命中无独立融合分 provenance。本 phase add-only `hybrid_score=17` 达成 v1 parity（融合分端到端携带、非由 score 推断，ADR-013）。
- **最 surgical 的「贯通而非重写」**：core hybrid 融合（`search_hybrid` + RRF）+ reranker 数据面 opt-in（`reranker_from_env`）+ `?semantic` 转发范式（task-20.1）+ `vector_score` add-only provenance 范式（task-32.3）+ proto add-only 演进（ADR-015 D1）均已存在，本 phase **0 backend 算法改动**，复用全部既有范式只加 proto 两字段 + 数据面 dispatch 分支 + Go 三处转发 + smoke 断言，无新机制、无新 dep。
