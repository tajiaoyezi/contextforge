## 2. 业务价值

第二轮治理债清扫——补齐「L2 缓存有界、memstore 访问序 LRU + 不变量、observability 索引重放 + trace 隔离、export 可配超时」四组真实债项，且经 grounding 对 survey 多处夸大据实诚实校正（这是本 phase 的 ADR-013 核心价值）：

### 33.1 L2 embedding 缓存有界（l2-embedding-cache-bound，🟢）

- `core/src/embedding/cache.rs` L1 自 Phase 31 已是 `BoundedCache` FIFO（`:23` `DEFAULT_EMBEDDING_CACHE_CAP=50_000`），但 **L2 SQLite `embedding_cache` 表无任何上限**：`sqlite_put`（:153-161）的 `INSERT OR REPLACE INTO embedding_cache`（:155）只写不删，行随长跑 opt-in SQLite 持久缓存只增不减。表经 `CREATE TABLE IF NOT EXISTS`（:110-120）建立（**非编号 migration**），列 `(content_hash, provider, dim, vector)`、PK `(content_hash, provider)`、含隐式 rowid。
- 本 phase 补 row-count cap + rowid-FIFO 驱逐：`sqlite_put` 写后 `COUNT(*)`，超 cap 则 `DELETE ... WHERE rowid IN (SELECT rowid FROM embedding_cache ORDER BY rowid ASC LIMIT overflow)`。用隐式 rowid → **0 新 dep、0 schema migration**；带 created_at 列的真 LRU 须 ALTER 既有用户文件 → honest-defer `[SPEC-DEFER:phase-future.l2-cache-true-lru]`。
- **HONEST CAVEAT（不夸大）**：`with_sqlite` 无生产调用点（test-only，`:331`/`:337`），出厂 daemon 走 memory-only L1 → 本项是 **opt-in 路径的防御性加固、非确证的 live leak**（如实陈述，不当作已发生的生产泄漏宣称）。L2 cap 默认 const 与 `DEFAULT_EMBEDDING_CACHE_CAP`（:23）并列；公共构造（`new`/`with_sqlite`/`with_capacity`）保源码兼容。镜像 TEST-31.2.1（`:345`）的驱逐测试形态。

### 33.2 memstore 访问序 LRU + hard-delete 不变量（memstore-lru-and-harddelete-invariant，🟢）

- **(a) console-api memstore FIFO → 访问序 LRU**：`internal/consoleapi/memstore.go` `cacheChunkUnlocked`（:76-91）/ `cacheTraceUnlocked`（:96-111）今为 FIFO——既有键覆盖**不**重排序、驱逐弹 `[0]`；读路径 `GetSourceChunk`（:341-352）/ `GetSearchTrace`（:357-368）**不** move-to-front。升级为命中即 move-to-front（读路径 + 既有键覆盖均重排），cap 沿用 `resolveCacheCapacity`（Phase 31，env `CONTEXTFORGE_CONSOLEAPI_CACHE_CAP`，默认 256）。既有 `TestMemStore_CacheEviction_FIFO`（`memstore_test.go:209-243`，硬断言 FIFO）须改为 LRU 断言；`types.go:69` / `handlers.go:277` 注释已写 'in-memory LRU'（术语已先行）。0-dep，fallback-mode only（生产接真实 backend）。
- **(b) memory hard-delete 级联 = HONEST-DEFER（non-issue，无可级联物）**：`core/src/memory/store.rs` `hard_delete`（:235-246）做单条 `DELETE FROM memory_items`。**完整 schema 审计**（6 表 migration 0010-0018）：`memory_id` **仅** `memory_items` 单表 PK；无任何其它表持有 `memory_id`；无 memory 向量/embedding 表（grep=0）；vector-store 层与 memory 零耦合。故无任何可悬挂物 → 写级联代码 = 投机/不可能场景（CLAUDE.md 简单优先）。交付物 = 一条**不变量测试**（schema 自省：`memory_items` 是唯一含 `memory_id` 的表 + `get(id)=None` after hard_delete），使未来若加 `memory_id` FK 会令其失败、被迫做真实决策。打 `[SPEC-DEFER:phase-future.memory-harddelete-cascade]`（仅当未来真加 FK）。
- **(c) `handleMemoryPin` strict-400 = DROPPED（刻意契约，不改）**：`handlers.go:521-549` 对 malformed/empty/absent body **刻意**回落 `pin=true` 以保 v0.7 宽松契约（doc-comment :519-524，task-17.1 / ADR-022 D2）。改 400 违 ADR-004 且与已 Accepted 的 ADR-022 D2 冲突。记为一项**诚实 non-change**（无代码改动）；在 ADR-038 D4 写明其刻意保持宽松。

### 33.3 observability 索引重放 + trace 隔离（observability-indexing-replay-and-trace-isolation，🟢 mapper/SQL · 🟡 e2e）

## 7. 阶段级风险

- **R1（中）L2 cap 驱逐误删 / 默认行为回归**：`sqlite_put` 加 `COUNT(*)` + `DELETE ... rowid` 若 overflow 计算或 rowid 序有偏，会误删非最旧行或破默认行为。
  - **缓解**：task-33.1 仅在超 cap 时删 `overflow = count - cap` 行、严格 `ORDER BY rowid ASC LIMIT overflow`（隐式 rowid 单调递增 = 插入序 = FIFO）；TEST-33.1.1 断言驱逐最旧、保留最新、COUNT 回 cap；默认 cap 足够大（与 L1 同量级），默认行为不变（既有 31.2.1 维持绿）。stop-condition：rowid-FIFO 误删 / 既有测试退化则 AC1 不标 `[x]`。
- **R2（中）memstore FIFO→LRU 改既有测试语义易过度**：把 `TestMemStore_CacheEviction_FIFO` 改 LRU 须改其断言而非删；过度改易掩盖回归。
  - **缓解**：task-33.2 仅把命中 move-to-front + 既有键覆盖重排接入，cap/eviction 机制不变（仍弹队首，但队首语义由「插入最旧」变「最久未用」）；改写测试保「插满→命中早项→插新项→被驱逐的是未命中的次旧项」LRU 断言。fallback-mode only（生产接真实 backend，不影响生产）。stop-condition：LRU 命中重排不生效 / 生产路径受影响则 AC2 不标 `[x]`。
- **R3（中）memory hard-delete 误实现投机级联**：survey 暗示 hard-delete 须级联，易诱导写跨表删除代码，但 schema 审计无可悬挂物。
  - **缓解**：task-33.2 据完整 schema 审计（`memory_id` 仅 `memory_items` 单表 PK、无其它表持有、无 memory 向量表）只交付**不变量测试**（schema 自省 + `get=None`），**不写任何级联实现**（CLAUDE.md 简单优先；ADR-013 不为不可能场景写代码）；级联记 `[SPEC-DEFER:phase-future.memory-harddelete-cascade]`（仅当未来真加 FK）。stop-condition：若写投机级联则越界——本 task 仅不变量测试。
- **R4（中→🟡）indexing 重放端到端须运行 daemon — 部分维度不可在 CI 默认闭环**：migration 0019 + emit 持久化 + replay mapper 可经单测/roundtrip 验，但重启重放端到端须 running daemon / job runner，CI 默认不跑。
  - **缓解**：task-33.3 以「migration 0019 + emit 持久化（roundtrip 🟢）+ replay mapper 纯函数单测（🟢，镜像 TEST-26.2.3）+ 重启 e2e 🟡 honest-defer `[SPEC-DEFER:phase-future.indexing-replay-e2e]`」拆分；AC3 以 mapper/roundtrip/SQL/handler PASS 满足，e2e 维度据真实 daemon 跑出后回填，不伪造（ADR-013）。stop-condition：e2e 维度未真实跑出则仅记 honest-defer，不标该维度达成。
- **R5（中）proto add-only `workspace_id` 不破既有 client + 空=聚合全部**：`GetSearchTraceRequest`/`ListQueriesRequest` 加 `workspace_id` 须 add-only（既有 field tag 不动）且空值保聚合全部，否则破既有控制面 client 或默认行为。
  - **缓解**：task-33.3 用下一可用 tag，既有 field tag/类型不动（ADR-004 既有契约不变）；TraceStore `WHERE workspace_id` 仅在非空时过滤、空 workspace_id 不加 WHERE（保聚合全部）；buf generate 重生 binding + 单测断言空 workspace 结果与改前一致。stop-condition：add-only field 破既有 client / 空 workspace 非聚合全部则 AC3 不标 `[x]`。
- **R6（低）drain-timeout 误重实现 + %v→%w 误改 non-bug**：survey 把已交付的 drain-timeout 与 non-bug 的 %v→%w 列为待办，易诱导重复实现 / 误改导致 vet-error。
  - **缓解**：task-33.3 drain-timeout **verify-only**（引用既有 `drainTimeoutFromEnv` + `TestDrainTimeoutFromEnv`，与 Phase 31 event-bus-partition verify-only 同形，不重实现）；task-33.4 `%v→%w` 据实记 non-bug 不改（`daemon/search.go` 无 fmt；`cli/search.go:88` 终端 Fprintf 用 `%w` 无效且未丢 grpc Status），在 §范围外 + ADR-038 D4 记录。stop-condition：若重实现 drain-timeout / 误改 %w 致 vet-error 则越界。
