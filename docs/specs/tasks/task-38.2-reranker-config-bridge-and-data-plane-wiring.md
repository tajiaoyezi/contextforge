## 1. Background

Phase 21（task-21.2，ADR-026）已确立 reranker 抽象：`core/src/rerank/` 的 `Reranker` trait（`traits.rs`）+ 决定性 `IdentityReranker`（`identity.rs`）默认 + feature-gated `CrossEncoderReranker`（`cross_encoder.rs`，`reranker-fastembed`），并在 `core/src/retriever/mod.rs:630` 提供 `with_reranker(Arc<dyn Reranker>)` 的 opt-in builder seam。但 reranker **至今从未由配置在生产热路径接线**——`with_reranker` 仅在 `retriever/mod.rs` 的 task-21.2 单测（`:1810` `test_21_2_2_with_reranker_seam_applies_and_none_unchanged`）里被调用过；生产路径 `server.rs` 的 hybrid（`wired` `:345`）与 semantic（`wired` `:390`）两处、`data_plane/search.rs` 的 semantic（`wired` `:285`）一处，都只 `.with_embedder().with_vector_searcher()`、**从不 `.with_reranker()`**。task-38.1（本 phase 同批）建 `RemoteRerankerProvider` + `select_reranker` 工厂 + `reranker-remote` feature + live harness，对真实远程 rerank 端点端到端验证 MRR / recall@1（SiliconFlow `/v1/rerank` / `Qwen3-VL-Reranker-8B`，remote-vs-identity 基线对照）。本 task 端到端兑现「配置 → 生产路径消费」两半：(A) 把 reranker 配置补进 Go 侧 `config.toml [reranker]` 段并经跨进程 env-bridge 过桥到 core（镜像 task-37.2 `setRemoteEnv` 的 `[remote]` 桥），(B) 在 Rust 数据面把 reranker 从配置在三处生产路径**首次 opt-in 接线**（新 `reranker_from_env()` 读 `CONTEXTFORGE_RERANKER_PROVIDER` → 非空/非 `none` 时 `select_reranker` → `with_reranker`）。它是 task-37.2 的 reranker 类比再加一段「数据面消费」：

- **B1 无 `[reranker]` config 段、无 reranker 的 config→core env 过桥**：`config.toml` 经 `internal/config/config.go` 解析，已有 `[remote]`（`RemoteProviderConfig{Enabled/Provider/Endpoint/Model}` `:50-61`）/ `[embedding]` / `[vector]` / `[[collections]]` 段，但**无 `[reranker]` 段**——用户在 `config.toml` 里没有声明 reranker endpoint / model / provider 的落点，且 reranker 配置当前完全未被桥接为 core 的 `CONTEXTFORGE_RERANKER_*` 环境变量。task-38.1 的 `select_reranker` `"remote"` arm 须 `CONTEXTFORGE_RERANKER_ENDPOINT` / `_MODEL` / `_PROVIDER`（镜像 `factory.rs:54-57` 的 `CONTEXTFORGE_REMOTE_*`），Go 侧须补段 + 过桥。
- **B2 reranker 从未由配置在生产路径接线（数据面债，NEW）**：`with_reranker` builder seam（`retriever/mod.rs:630`）自 task-21.2 起仅 opt-in、仅测试用——`server.rs` hybrid/semantic 与 `data_plane/search.rs` semantic 三处生产 `wired` 链**从不调 `.with_reranker()`**，故即便配置了 reranker 也不生效。因此「只加 Go `[reranker]` config 桥而数据面不消费」不诚实（ADR-013）——本 task 必须端到端：Go 桥 + Rust 数据面在三处生产接线点消费。新 `reranker_from_env() -> Result<Option<Arc<dyn Reranker>>, RerankError>` 读 `CONTEXTFORGE_RERANKER_PROVIDER`，空 / `"none"` → `None`（默认无 rerank，向后兼容），否则 `select_reranker(name)` 包 `Some`；三处生产路径 build `wired` 后 `if let Some(rr) = reranker_from_env()? { wired = wired.with_reranker(rr) }`。
- **B3 Rust core 无 toml dep（不能在 Rust 侧读 config.toml）**：Rust core 持 `serde` / `serde_json` 但**无 `toml` 依赖**——在 Rust 侧加 config.toml reader 须引 `toml` crate，破 0 新 dep（ADR-008）。故配置桥接须在已解析 `config.toml` 的 **Go 侧**完成；core 经 task-38.1 的 `select_reranker` env 读取路径接收（本 task 的 Rust 改动限于新 `reranker_from_env()` 包装 + 三处生产接线，不引 toml dep）。
- **B4 已证的跨进程 env-bridge 范式（`CONTEXTFORGE_REMOTE_*` / `CONTEXTFORGE_VECTOR_*` / `CONTEXTFORGE_DATA_DIR`）**：`cmd/contextforge/main.go:340-385` 的 `setRemoteEnv`（task-37.2）与 `:288-338` 的 `setVectorEnv`（task-34.2）把配置经 `os.Setenv(...)` 导出为父进程环境变量；spawned core daemon 经 `internal/daemon/daemon.go` `launch` 的 `exec.Command`（继承父进程 env）启动，于 Rust 侧经 env 读取等价路径拾取。本 task **复用同一已证范式**——加一个 `setRerankerEnv`（精确镜像 `setRemoteEnv`），把 `[reranker]` 段桥接为 `CONTEXTFORGE_RERANKER_ENDPOINT` / `_MODEL` / `_PROVIDER`。**这是同一跨进程 env-bridge，不是延后的 `daemon.Options` 字段重构**（后者 [SPEC-DEFER:phase-future.daemon-options-datadir]）。
- **B5 env-wins 向后兼容 + 无 `[reranker]` 段 / unset = 默认无 rerank**：显式设置的环境变量**优先于** config 文件（与 `setRemoteEnv` / `setVectorEnv` 的「env 已设则不覆盖」语义一致）——保证既有 env-only 工作流向后兼容。无 `[reranker]` 段（或字段为空）⇒ 不导出任何变量 ⇒ `CONTEXTFORGE_RERANKER_PROVIDER` 保持调用方原状（默认 unset）⇒ `reranker_from_env()` 返回 `None` ⇒ 三处生产路径不调 `with_reranker` ⇒ **检索结果字节等价于当前无 rerank 行为**（ADR-004，reranker 仍 opt-in）。
- **B6 安全红线：API KEY 永不进 config.toml，仅 env**：`CONTEXTFORGE_RERANKER_API_KEY` 是密钥，**永不**写入 `config.toml`、**永不**由 `setRerankerEnv` 处理或导出——`RerankerConfig` **不含 api-key 字段**（精确镜像 `RemoteProviderConfig` 不含密钥），`setRerankerEnv` 只过桥非密钥的 endpoint / model / provider；密钥仅由用户显式设 `CONTEXTFORGE_RERANKER_API_KEY` 环境变量，core 经 task-38.1 的 provider 读取且永不记录（PRD 安全基线 / ADR-004 opt-in / ADR-013 据实记安全红线）。

本 task 为 code-local 🟢 可验证（Go config TOML round-trip + `setRerankerEnv` env-wins 单测 + Rust 数据面 opt-in 接线 `identity` provenance marker 单测）；Go 侧 0 新 dep（沿用 task-1.2 手写 TOML codec + 既有 `os` / `errors` 标准库）；Rust core 依赖面零变化（不引 toml dep，仅加 `reranker_from_env()` 包装 + 三处接线，复用 task-38.1 的 `select_reranker`）。

### 5.2 关键设计 — Go [reranker] 段 add-only + setRerankerEnv 跨进程 env-bridge + Rust 数据面 opt-in 接线（env-wins / API key env-only / Rust 0-dep / 默认字节等价无 rerank）

- **B1 Go `[reranker]` 段 add-only（镜像 `[remote]`）**：新 `RerankerConfig{Enabled bool / Provider string / Endpoint string / Model string}`（toml 标签 `enabled`/`provider`/`endpoint`/`model`，**无 api-key 字段**），挂到 `Config.Reranker`（与 `Remote` `:38` 并列）；`encodeTOML`（`:201-226`）加 `\n[reranker]\n` 段输出 `enabled`/`provider`/`endpoint`/`model`（镜像 `[remote]` `:208-212`，置于 `[remote]` 段后、`[embedding]` 段前）；`decodeTOML`（`:228-293`）加 `case line == "[reranker]"`（`section = "reranker"`，镜像 `:238-240`）+ `case "reranker"`（分派 `assignReranker`，镜像 `:268-271`）；新 `assignReranker`（镜像 `assignRemote` `:327-355`：`enabled`→`ParseBool`，其余→`parseTOMLString`）。**absent `[reranker]` 段 ⇒ `Config.Reranker` 为 zero value**，与既有缺段行为一致（手写 codec 容忍缺段/缺字段）。
- **B2 Rust 数据面 opt-in 接线（NEW — reranker 生产路径首次接线）**：加 `reranker_from_env() -> Result<Option<Arc<dyn Reranker>>, RerankError>`——读 `CONTEXTFORGE_RERANKER_PROVIDER`，trim 后空 / 等于 `"none"`（不分大小写）⇒ `Ok(None)`（默认无 rerank）；否则 `select_reranker(name).map(Some)`（task-38.1 工厂）。`server.rs` hybrid `wired`（`:345`）+ semantic `wired`（`:390`）+ `data_plane/search.rs` semantic `wired`（`:285`）三处 build `wired`（改 `let mut wired = ...`）后插：`if let Some(rr) = reranker_from_env().map_err(|e| Status::internal(format!("reranker: {e}")))? { wired = wired.with_reranker(rr); }`。`with_reranker`（`retriever/mod.rs:630`）已存在、task-21.2 已测 seam 应用语义（`:1810`），本 task 在生产路径首次消费——`search_semantic` / `search_hybrid` 内部经 reranker re-order top-k（reranker 注 provenance reason，见 `IDENTITY_RERANK_REASON` / `CROSS_ENCODER_RERANK_REASON`）。
- **B3 setRerankerEnv 跨进程 env-bridge（镜像 setRemoteEnv）**：在 `cmd/contextforge/main.go` 加 `setRerankerEnv(dataDir string) func()`，best-effort `config.Load(dataDir)`（load 失败非致命；missing config 静默经 `errors.Is(err, os.ErrNotExist)` 守护；real parse/read 失败 stderr WARN，镜像 `setRemoteEnv`）；逐变量复用 `setIfAbsent`（env-wins 守卫）——对 `CONTEXTFORGE_RERANKER_ENDPOINT`（取 `cfg.Reranker.Endpoint`）/ `_MODEL`（`cfg.Reranker.Model`）/ `_PROVIDER`（`cfg.Reranker.Provider`）：**仅当值非空且 env 未显式设（`!had`）时** `os.Setenv`；返回 restore 闭包（恢复每个变量的旧值 / unset，镜像 `setRemoteEnv` 返回 `func()`）。spawned core daemon 经 `daemon.go` `exec.Command`（继承父进程 env）启动 → Rust 侧 `reranker_from_env` + `select_reranker` 拾取，与 `CONTEXTFORGE_REMOTE_*` / `CONTEXTFORGE_VECTOR_*` 同跨进程范式。接线 `doServe`（`:112` `setRemoteEnv` 后）+ `doMCP`（`:150` `setRemoteEnv` 后）。
- **B4 env-wins（向后兼容）**：`setRerankerEnv` 对每个变量先 `os.LookupEnv` 探测；**若该变量已显式设置（`had == true`）则不 `os.Setenv` 覆盖**——显式 env 优先于 config 文件（既有 env-only 工作流向后兼容）。这是与 `setRemoteEnv` / `setVectorEnv` `setIfAbsent`（`!had` 才 Setenv）一致的「env-wins」实现。
- **B5 安全红线：API KEY 永不进 config.toml / 仅 env**：`RerankerConfig` **不含** api-key 字段（精确镜像 `RemoteProviderConfig` 无密钥字段）；`encodeTOML` 永不写 API key；`config.toml` 永不含密钥；`setRerankerEnv` **只过桥** `CONTEXTFORGE_RERANKER_ENDPOINT` / `_MODEL` / `_PROVIDER` 三个非密钥变量、**绝不处理 / 绝不导出 `CONTEXTFORGE_RERANKER_API_KEY`**。API key 仅由用户显式设 `CONTEXTFORGE_RERANKER_API_KEY` env，core 经 task-38.1 provider 读取且永不记录。PRD 安全基线 / ADR-004 opt-in / ADR-013 据实记安全红线（[SPEC-DEFER:phase-future.remote-api-key-secret-store] 专用 secret-store 不在本 task）。
- **无 `[reranker]` 段 / unset = 默认无 rerank 字节等价（ADR-004）**：absent `[reranker]` 段 / 字段空 ⇒ `cfg.Reranker.{Endpoint,Model,Provider}` 为 `""` ⇒ `setRerankerEnv` 三变量均不 `os.Setenv`（`val == ""` 守卫）⇒ `CONTEXTFORGE_RERANKER_PROVIDER` 保持调用方原状（默认 unset）⇒ `reranker_from_env()` 返回 `None` ⇒ 三处生产路径不调 `with_reranker` ⇒ **检索结果字节等价于当前无 rerank 行为**。默认 build / 默认运行行为不变（reranker 仍 opt-in，须显式 `CONTEXTFORGE_RERANKER_PROVIDER` 非空非 none 才激活）。
- **feature-off 显式 Err 不静默（ADR-013）**：`CONTEXTFORGE_RERANKER_PROVIDER=remote`（或 `cross-encoder`）但对应 feature（`reranker-remote` / `reranker-fastembed`）未编译时，`select_reranker` 返回显式 `RerankError`（task-38.1 工厂语义，镜像 `factory.rs:44-46` / `:70-72`）→ `reranker_from_env()?` 传播 → 生产路径 `Status::internal(format!("reranker: {e}"))`，**绝不静默 fallback 到无 rerank 或 identity**（misconfiguration 显式失败，便于诊断）。
- **Rust core 0 toml dep（ADR-008）**：配置桥接全程在 Go 侧（已解析 `config.toml`）+ 环境变量过桥完成；Rust core **不引 `toml` crate**（依赖面零变化）；本 task 的 Rust 改动限于新 `reranker_from_env()` 包装（读 std `env`）+ 三处生产 `wired` 接线 + `with_reranker` 既有 seam，复用 task-38.1 的 `select_reranker`（不重复实现 provider）。
- **非 `daemon.Options` 字段重构**：本 task 复用已证的跨进程 env-bridge（同 `CONTEXTFORGE_REMOTE_*` / `CONTEXTFORGE_VECTOR_*` / `CONTEXTFORGE_DATA_DIR`），**不**引入 `daemon.Options` 结构字段把配置经 `daemon.Options` 传入 [SPEC-DEFER:phase-future.daemon-options-datadir]（该重构属另一延后边界，本 task 不依赖、不实现，ADR-013 据实声明用已证范式而非新建结构）。

## 8. Risks

- **R1（中）`[reranker]` 段加入破既有 config.toml round-trip**：加 `[reranker]` 段输出 / 解析可能影响既有 `[remote]`/`[embedding]`/`[vector]`/`[[collections]]` 段的编解码顺序或解析。
  - **缓解**：`[reranker]` 段紧随 `[remote]` 段（`encodeTOML` `:212` 后）输出、置于 `[embedding]` 段前，`decodeTOML` 加独立 `case line == "[reranker]"` + `case "reranker"`（不改既有 `case`），`assignReranker` 独立 helper；TEST-38.2.1 断言既有段 round-trip 不退化 + 既有 TEST-1.2.* / TEST-34.2.* / TEST-37.2.* 全绿。stop-condition：既有段 round-trip 单测退化则 AC1 不标 `[x]`。
- **R2（中）旧 config.toml（无 `[reranker]` 段）兼容**：既有用户的 `config.toml` 无 `[reranker]` 段，Load 须得 zero value（不报错）；Save 后新增 `[reranker]` 段（字段全 zero）须语义等同 unset。
  - **缓解**：手写 codec 缺段 ⇒ `Config.Reranker` zero value（与既有缺段一致）；Save 出的字段 zero（空字符串）经 `setRerankerEnv` 守卫（`val != ""` 才导出）⇒ 不导出 ⇒ 默认无 rerank。TEST-38.2.1 含「不含 `[reranker]` 段 legacy Load 得 zero value」断言。
- **R3（高）数据面 opt-in 接线破默认无 rerank 字节等价（向后兼容）**：若三处生产路径无条件 `with_reranker`（即便 `reranker_from_env()` 返回 `None`），会改变默认（unset）检索结果，破当前行为。
  - **缓解**：三处生产路径用 `if let Some(rr) = reranker_from_env()? { wired = wired.with_reranker(rr) }`——**None 分支完全不改 `wired`**（仅 `let` → `let mut`）；TEST-38.2.3 含「unset ⇒ 无 marker、向后兼容字节等价无 rerank」断言。stop-condition：unset 向后兼容单测退化则 AC2 不标 `[x]`。
- **R4（中）env-wins 语义被误实现为 config-wins**：若 `setRerankerEnv` 无条件 `os.Setenv` 会让 config 覆盖已显式设的 env（破向后兼容）。
  - **缓解**：`setRerankerEnv` 对每个变量先 `os.LookupEnv`，仅 `!had` 时 `os.Setenv`（复用 `setRemoteEnv` `setIfAbsent` 范式）；TEST-38.2.2 含「env 已显式设 ⇒ 不被 config 覆盖」断言（env-wins）。stop-condition：env-wins 单测不过则 AC1 不标 `[x]`。
- **R5（高）API key 误进 config.toml / 误被 setRerankerEnv 导出（安全红线破）**：若给 `RerankerConfig` 加 api-key 字段或让 `setRerankerEnv` 处理 `CONTEXTFORGE_RERANKER_API_KEY`，密钥会落盘 / 被 Go 侧搬运，破 PRD 安全基线。
  - **缓解**：`RerankerConfig` **不含** api-key 字段（`encodeTOML` 永不写密钥）；`setRerankerEnv` 仅过桥 endpoint/model/provider 三个非密钥变量、**绝不**触及 `CONTEXTFORGE_RERANKER_API_KEY`；TEST-38.2.2 含「`CONTEXTFORGE_RERANKER_API_KEY` 永不被 `setRerankerEnv` 导出 / 处理」断言（设一个哨兵 env 值，调用后断言其未被 `setRerankerEnv` 改动且无新 API-key env 被设）。stop-condition：API-key-never-exported 单测不过则 AC1 不标 `[x]`。
- **R6（中）feature-off 静默 fallback**：若 `CONTEXTFORGE_RERANKER_PROVIDER=remote` 但 `reranker-remote` feature 未编译，静默退回无 rerank / identity 会掩盖 misconfiguration。
  - **缓解**：`select_reranker`（task-38.1）feature-off 显式 `RerankError`（镜像 `factory.rs:44-46`/`:70-72`）→ `reranker_from_env()?` 传播 → 生产路径 `Status::internal`，绝不静默 fallback（ADR-013）；TEST-38.2.3 含「feature-off provider ⇒ 显式 Err」断言（按可编译范围验证 `cross-encoder` feature-off 路径）。
- **R7（低）测试改进程全局 env 致并行测试串扰**：`setRerankerEnv` / `reranker_from_env` 测试 `os.Setenv` / `std::env::set_var` 改进程全局环境变量，并行测试可能相互干扰。
  - **缓解**：Go 测试用 `t.Setenv`（自动恢复 + 标记不可 `t.Parallel`）或显式 restore 闭包；Rust 测试串行化 env 段（同既有 env-gated 测试惯例）/ 断言后还原；与既有 `setRemoteEnv` / `setVectorEnv` 测试同惯例。
- **R8（低）跨进程 env-bridge 被误读为新装配机制**：env-bridge 易被误读为新建的 daemon 配置传递机制，而非复用 `CONTEXTFORGE_REMOTE_*` / `CONTEXTFORGE_VECTOR_*` 已证范式。
  - **缓解**：spec §1 B4 / §5.2 B3 / §5.3 据实记「同 `CONTEXTFORGE_REMOTE_*`/`CONTEXTFORGE_VECTOR_*`/`CONTEXTFORGE_DATA_DIR` 已证范式，非 `daemon.Options` 字段重构 [SPEC-DEFER:phase-future.daemon-options-datadir]」（ADR-013 不夸大）；Rust core 0 toml dep 保依赖面零变化。
