# Phase 18 spike — `qdrant`

> Backend: `qdrant-client` 1.18.0 → external **Qdrant server v1.18.1** (static musl binary, gRPC
> 6334). Generated by `contextforge-bench` (task-18.2 harness). **Real recall data on Linux x86_64.**
> Unlike hnsw / sqlite-vec, Qdrant is an **out-of-process server** (`is_local() == false`).

### RSS measurement note (read this — the harness RSS is NOT qdrant's)

The harness `sample_rss_mb()` samples the **bench client process** (idle 7.0 MB / index 10.78 MB) —
this is the gRPC *client*, **not** the Qdrant server, so it does not represent Qdrant's memory.
Qdrant's actual footprint was sampled out-of-band from the server process: **VmRSS ≈ 104.8 MB**
(`ps -C qdrant`, with the 5000-vector `spike` collection loaded). That is **~10× the in-process
backends** (hnsw 11 MB / sqlite-vec 8.5 MB at the same n) — the resident cost of running a separate
database server. A truly empty Qdrant baseline is in the tens of MB; the 104.8 MB figure is the
indexed-state server RSS.

Cross-check — `noop` baseline, same params: recall@5/10 = 0.0 (empty hits) — the harness measures the
backend. recall=1.0 on synthetic is near-ideal; representative ranking needs dogfood + larger n.

## Environment

- platform / arch: `x86_64-unknown-linux-gnu` (WSL2 Ubuntu 26.04)
- Qdrant server: v1.18.1 `qdrant-x86_64-unknown-linux-musl` static binary, gRPC 6334 / REST 6333,
  launched as a background process (no Docker — Docker Desktop's WSL integration is not wired to this
  distro, so the native binary is used instead)
- toolchain: rustc 1.96.0, release profile; qdrant-client 1.18.0 (async, bridged to the sync trait
  via an owned current-thread tokio runtime + `block_on`)
- embedding source: deterministic seed vectors; `Distance::Cosine` → KNN == the harness cosine truth

## Trade-off discussion

- **deployment model**: **external server** (`is_local() == false`) — the defining difference vs every
  other candidate. Requires running, provisioning, and lifecycle-managing a Qdrant process (or
  cluster). Qdrant has no native Rust embedded mode. This is operational surface the in-process
  backends (hnsw / sqlite-vec) do not have.
- **memory**: ~104.8 MB resident server RSS with 5000 vectors — ~10× the in-process backends. At the
  100k+ target Qdrant's server overhead amortizes better, but the floor is high for small/edge
  deployments.
- **latency**: P95 0.650 ms includes a localhost gRPC round-trip, so it is higher than the in-process
  backends (hnsw 0.382 ms / sqlite-vec 0.167 ms) at n=5000 despite Qdrant's HNSW index. Over a real
  network the round-trip cost grows; in-process backends have none.
- **build / compile complexity**: `qdrant-client` pulls tonic/prost (already in-tree) + reqwest/
  hyper-rustls; compiles cleanly (~35 s release). No C/C++ toolchain needed on the client side.
- **capability / scale**: the only candidate built for **distributed scale, replication, filtering,
  quantization, and multi-tenancy** out of the box. For a single-binary local-first tool this is
  largely unused weight; for a hosted/multi-agent deployment it is the natural fit.
- **persistence**: server-managed on-disk storage (durable, snapshots, WAL) — no application-side
  persistence code, unlike hnsw (needs its own serialization).
- **verdict**: **viable and feature-rich, but heavyweight** for the local-first single-binary profile.
  Excellent recall and capability; the cost is an external server (~105 MB, lifecycle, gRPC latency).
  The selection (task-18.7) must weigh ContextForge's deployment target: in-process simplicity vs
  Qdrant's scale/feature ceiling.
