# Phase 18 spike — `sqlite-vec`

> Backend: `sqlite-vec` 0.1.9 (asg017/sqlite-vec `vec0` SQLite extension), registered into rusqlite's
> bundled SQLite via `sqlite3_auto_extension`. Generated by `contextforge-bench` (task-18.2 harness).
> **Real recall data collected on Linux x86_64 (gcc 15.2.0)** — phase-18 §7 R1 P0 platform.

random vectors); the representative recall ranking vs other backends must come from the dogfood
corpus + a Linux release run at the 100k target (task-18.7).

## Environment

- platform / arch: `x86_64-unknown-linux-gnu` (WSL2 Ubuntu 26.04)
- toolchain: rustc 1.96.0, release profile; C extension compiled by gcc 15.2.0 via `cc-rs`
- embedding source: deterministic seed vectors (unit-normalized; the `vec0` default L2 distance is
  monotonic with cosine for unit vectors, so `vec0` KNN == the harness's cosine ground truth)
- storage: in-memory SQLite (`Connection::open_in_memory`), embeddings inserted as JSON text

## Trade-off discussion

- **build / compile complexity**: compiles via gcc/clang on Linux (P0) in ~11s. Reuses the existing
  `rusqlite` (bundled SQLite) dependency — the extension is registered with
  `sqlite3_auto_extension(sqlite3_vec_init)`, no second SQLite. **Caveat**: the C `vec0` amalgamation
  does **not** build under Windows MSVC (`cl.exe` is not exercised by the project upstream) — see the
  Windows note below. The `0.1.10-alpha.4` pre-release also ships an incomplete amalgamation (missing
  `sqlite-vec-diskann.c`); this spike pins **stable `=0.1.9`**.
- **persistence model**: `vec0` is a real SQLite virtual table, so on-disk persistence is native
  (open a file-backed `Connection` instead of `:memory:`) — unlike hnsw, which needs its own graph
  serialization. This spike used an in-memory DB to measure cold-start/reindex; a file-backed
  deployment aligns naturally with ADR-002's SQLite+Tantivy persistence model.
- **search model**: `vec0` KNN is an exact brute-force scan (not ANN), so cold-start is ~23× faster
  than hnsw's graph build (37ms vs 836ms at n=5000) and P95 is lower at this n (0.167ms vs 0.382ms).
  The trade-off inverts at scale: exact scan is O(n) per query, so at the 100k+ target hnsw's ANN
  graph should win on latency. The n=5000 numbers here are not representative of the 100k regime.
- **API stability / ecosystem maturity**: sqlite-vec 0.1.9 is the latest stable; pinned in
  Cargo.lock. Mature SQLite host; the extension itself is young (0.1.x) but actively developed.
- **verdict**: **viable on Linux (P0)** with native on-disk persistence and excellent small-n
  latency; **not viable on Windows MSVC** (dev-box build-blocked). The exact-scan model favors
  correctness and persistence over large-n latency — the representative ranking vs hnsw at the 100k
  target is the task-18.7 decision.

## Windows MSVC note (build-blocked on the dev box)

The `vec0` C extension does not compile under MSVC `cl.exe` (Chinese-localized diagnostics, exit 2).
This is platform-specific to the dev box, not a blocker for the Linux P0 target or CI. The
`vector-sqlite` feature is off by default, so the default build (including Windows) is unaffected —
0 new dependency at default features. `[SPEC-DEFER:phase-future.sqlite-vec-cross-platform]` tracks a
Windows MSVC port if ever needed.
