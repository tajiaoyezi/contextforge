# Phase 18 spike — `lancedb`

> Backend: `lancedb` 0.30.0 (embedded Lance columnar vector store, DataFusion query engine).
> Generated by `contextforge-bench` (task-18.2 harness). **Real recall data on Linux x86_64.**
> In-process (`is_local() == true`) and disk-backed (a Lance dataset).

Cross-check — `noop` baseline, same params: recall@5/10 = 0.0 (empty hits) — the harness measures the
backend. recall=1.0 on synthetic is near-ideal; representative ranking needs dogfood + larger n. At
n=5000 Lance does a flat (exact) scan; an IVF/HNSW index would change the latency profile at scale.

## Environment

- platform / arch: `x86_64-unknown-linux-gnu` (WSL2 Ubuntu 26.04)
- toolchain: rustc 1.96.0, release profile; **build needs `protoc`** (lance `build.rs`) — vendored
  protoc v35.0 via `PROTOC` env + cmake 4.2.3. First build of the Lance/DataFusion/Arrow stack ~5 min.
- arrow versions: Lance v7 uses **arrow 58**, datafusion 53 uses arrow 53 — both resolve in the tree.
  `arrow-array` is pinned to `58` to match Lance (lancedb re-exports only `arrow_schema`).
- async `lancedb` bridged to the sync trait via an owned current-thread tokio runtime + `block_on`.
- storage: on-disk Lance dataset under a temp dir; `DistanceType::Cosine` → KNN == the harness truth.

## Trade-off discussion

- **deployment model**: **embedded + disk-backed** — the only candidate that is both in-process (no
  server, unlike qdrant) *and* persisted as a real on-disk dataset (unlike hnsw's in-memory graph).
  This matches ContextForge's local-first single-binary profile while giving durable storage for free.
- **memory**: idle 30.5 MB / index 50.9 MB — the **heaviest in-process** footprint (the Lance +
  DataFusion runtime), ~5× sqlite-vec (8.5 MB) and hnsw (11 MB), but still far below qdrant's ~105 MB
  server.
- **latency**: P95 1.551 ms — highest of the in-process backends (DataFusion plans + executes each
  query), vs sqlite-vec 0.167 ms and hnsw 0.382 ms. A built vector index (IVF_PQ/HNSW) would help at
  scale; at n=5000 the flat scan + query overhead dominates.
- **write speed**: cold-start 7.4 ms / reindex 5.8 ms — the **fastest** of all four (columnar batch
  append is very cheap), vs hnsw's 836 ms graph build. Re-indexing is essentially free.
- **build / compile complexity**: the **heaviest** — pulls Lance + DataFusion + Arrow (hundreds of
  crates, ~5 min first build) and **requires `protoc`** at build time. The `vector-lancedb` feature is
  off by default so default/CI builds are unaffected, but enabling it adds real build-prereq surface.
- **capability**: columnar storage with SQL (DataFusion), full-text + vector + scalar filtering, and
  versioned datasets — a richer data model than a pure ANN index, useful if ContextForge wants
  combined metadata-filtered vector search on disk.
- **verdict**: **viable and a strong fit for local-first + persistence**, at the cost of the heaviest
  in-process memory, highest per-query latency at small n, and a `protoc`/Arrow-heavy build. The
  selection (task-18.7) weighs its embedded-durable model + fast writes against sqlite-vec's lighter
  footprint and lower latency.
