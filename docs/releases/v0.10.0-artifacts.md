## 7. Trade-offs ship 矩阵 (Phase 17)

| Decision | Decision rationale | Cost / Future work |
|---|---|---|
| `bool` type for IsPinned (ADR-022 D1) | Pin state always defined; `*bool` + omitempty would create render ambiguity | None — locked in ADR; future `pinned_at` timestamp `[SPEC-DEFER:phase-future.memory-pinned-at-timestamp]` |
| No `pinned_at` / `pin_actor` field in this amendment | Strict scope discipline; ADR-022 only adds the minimum needed for sort + icon UI | `pinned_at` / `pin_actor` `[SPEC-DEFER:phase-future.memory-pinned-at-timestamp]` / `[SPEC-DEFER:phase-future.memory-pin-actor]` |
| handleMemoryPin empty-body defaults to `pin=true` | Preserves v0.7-v0.9 callers; pointer-typed body cleanly distinguishes absent from explicit false | Lenient contract — malformed JSON also falls back to true. Could tighten to 400 if needed (`[SPEC-DEFER:phase-future.handle-memory-pin-strict-body]`) |
| Smoke v8 step 28 gated `MODE=real && sqlite3` | Runtime end-to-end needs both Rust daemon SQLite + sqlite3 CLI for seeding | LOCAL_ONLY/docker modes verify via unit tests instead (acceptable trade-off) |
| Skip migration 0017 (column already in 0013 from task-13.1) | Avoid `duplicate column name` on existing DBs; no actual migration needed | Documented as spec drift in PR #118 + task-17.1 §3 |
| MemMemoryStore fixture-1 preset IsPinned: true (ADR-022 D3) | At least one pinned fixture for Console UI fallback testing | Fallback mode opt-in only (ADR-018 deny default); no production impact |
| Cross-repo coordination via user-forwarded merge SHA | Per ADR-022 D5 + ADR-011 single-driver; ContextForge agent doesn't directly write Console repo | User has bridge role for each amendment ADR; pattern is reusable |
| No new `linux/arm64` image in v0.10 | v0.10 is schema amendment ship; image build infra unchanged from v0.9 | arm64 `[SPEC-DEFER:phase-future.multi-arch-image]` retained |

## 9. Future amendment template (ADR-022 D4/D5 pattern reusable)

The successful Phase 17 amendment establishes a template for future schema evolutions under ADR-015 D1 add-only constraint:

```
Step 1 (ContextForge): Author amendment ADR + Phase + task spec scaffold; ship as one E1 PR
                       (Status: Proposed for ADR; Pending for Phase + task spec)
Step 2 (User):         Forward Console-side amend prompt to Console main agent
Step 3 (Console):      Add field/type to Console contractv1.go as add-only;
                       merge to Console master (Console-first ship per D4)
Step 4 (User):         Forward Console PR merge SHA back to ContextForge main agent
Step 5 (ContextForge): Verify Console master HEAD contains expected change via gh api;
                       flip Phase + task Status: Pending → Ready → Done;
                       ship implementation PR with proto + Rust + Go + tests + smoke
Step 6 (ContextForge): Open closeout PR — ADR Proposed → Accepted + spec §10 final fills +
                       release notes + evidence + artifacts files
Step 7 (User):         After closeout PR merge, push annotated `v*` tag → release.yml
                       triggers GHCR image build + push + GitHub Release
Step 8 (User):         Notify Console team of new ContextForge release;
                       Console UI single-driver PR ships visual closure feature
```

Estimated total elapsed: 1-2 days for trivial single-field amendment; longer if cross-repo review cycle introduces back-and-forth. Phase 17 itself ran in ~1 day end-to-end (2026-05-28 scaffolding → 2026-05-28 closeout).
