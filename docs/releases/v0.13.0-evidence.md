## 3. 验证证据

### 1. 真实 SemanticRecall@K（task-20.2, real `FastEmbedProvider` all-MiniLM-L6-v2 dim 384, exact cosine, 经生产 `Retriever::search_semantic` 热路径, WSL2 Ubuntu / rustc stable）

| metric | task-20.2（生产 Retriever, 175 production chunks, uncapped） | task-19.5（standalone backend, 40 capped chunks）基线 | gate (≥0.70) |
|---|---|---|---|
| SemanticRecall@5 | **0.9667** (29/30) | 0.8333 | **PASS** |
| SemanticRecall@10 | **1.0000** (30/30) | 0.9333 | **PASS** |
| top-1 accuracy | **0.7333** | 0.60 | — |
| MRR | **0.8367** | 0.70 | — |

per-category（n=5 each）：config-location / error-reproduction / historical-decision / code-location / agent-memory-rule 全 recall@5=@10=1.0；log-troubleshooting @5=0.80 / @10=1.0。

**诚实 caveat（ADR-013）**：经生产 chunker（**uncapped**，11 files → **175 production chunks**）跑出的 recall@10=1.0 **部分含 file-level 膨胀**——chunk/file 多 → 「任一 chunk 命中 top-K」机械变易，正是 task-19.5 用 `MAX_CHUNKS_PER_FILE` 刻意抑制的 artifact。但区分性 top-1=0.7333 / MRR=0.8367（**高于** task-19.5 的 0.60/0.70，且不受 chunk 数膨胀影响——top-1 是单一最佳命中）证实真实路径表现，**非纯膨胀**。两口径方法不同（生产 chunker vs 受控窗口）**不直接可比**，但均过 ADR-006 A1 gate（`SemanticRecall@10 ≥ 0.70`）；task-20.2 为**代表性**测量（生产管道），task-19.5 为**受控区分度地板**。deterministic embeddings 仅证 plumbing 非召回质量（真实数值只来自 fastembed run）。数据源 + 方法 + ADR-013 三态声明见 `docs/spikes/phase-20-recall-via-retriever.md`；复跑 `cargo run -p contextforge-core --example phase20_recall_via_retriever --features embedding-fastembed`。
