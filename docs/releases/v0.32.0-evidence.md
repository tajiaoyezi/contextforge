### 3. rerank-forward 重界定（ADR-013 据实，MUST 记录）

**(a) reranker 保持服务端 env 驱动（ADR-043 D3 不变）**：reranker 由 `CONTEXTFORGE_RERANKER_PROVIDER` env 在服务端 opt-in（unset / "" / "none" → 无 rerank 字节等价）。Phase 39 **不**加 `?rerank` per-request 参数——`grpcclient` 无 rerank 字段转发（TEST-39.2.2 断言）。

**(b) `?rerank=true` per-request 控制 superseded（不实现，非缺口）**：历史 `[SPEC-DEFER:phase-future.console-api-rerank-forward]`（task-21.2 / task-21.3 记录）设想 `?rerank=true` per-request 转发，但 Phase 38（ADR-043 D3）此后确立 reranker 服务端 env 驱动——per-request 转发会让对外请求覆盖服务端 reranker 选型、破坏 env 驱动模型，故据实记为**被 D3 取代（superseded）、不实现**（ADR-013：不实现与既定设计冲突的控制项、也不伪装它已实现）。

**(c) 改交付 rerank provenance 可见性（fulfilled）**：rerank `reason` 标记经既有链路（Rust `SearchResult.reason` → proto `reason=14` → `grpcclient.protoToSearchResult` `Reason: p.Reason` → `contractv1.SearchResult.Reason` → REST JSON）端到端流通；Phase 39 加 TEST-39.2.2 断言 `reason`（`reranked:identity`）经 `protoToSearchResult` 映射可见 + smoke step 48 文档化——reranker env opt-in 时用户经对外 `POST /v1/search` 能看到结果被 rerank 过 + 服务端 reranker 配置生效。这是 `console-api-rerank-forward` 维度按 provenance-visibility 口径的真实兑现。
