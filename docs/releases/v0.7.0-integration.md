### 1. Daemon 部署形态

**完整启动命令**（需启动 2 个进程，cross-process gRPC bridge per ADR-016）：

```bash
# 进程 1: Rust 数据面 daemon (gRPC server)
./contextforge-core 127.0.0.1:50551 /var/lib/contextforge

# 进程 2: Go 控制面 REST proxy (Console contract v1)
./contextforge console-api-serve \
  --addr 127.0.0.1:48181 \
  --grpc-addr 127.0.0.1:50551 \
  [--auth-token <bearer>]   # 可选；空 = trusted-network mode
  [--fallback-inmem]        # 可选；gRPC 不可达时降级到 in-memory MemStore
```

| 项 | 默认值 | 备注 |
|---|---|---|
| REST `addr` | `127.0.0.1:48181` | Docker 模式自动改为 `0.0.0.0:48181`（compose mesh 可达） |
| gRPC `addr` | `127.0.0.1:50551` | Rust daemon 第 1 个 positional arg；console-api-serve `--grpc-addr` |
| Scheme | **http only** | v0.7 未实现 TLS（参见 §3 mTLS 段） |
| 数据目录 | `$CONTEXTFORGE_DATA_DIR` / Rust daemon 第 2 个 positional arg | 包含 `workspaces.db` / `jobs.db` / `memory.db` / `eval.db` / `index/` Tantivy / `imports/` |

**运行时依赖**：
- SQLite — 内嵌于 Rust binary，**无外部 sqlite-server 进程**
- **不需要** Redis / Postgres / Kafka / 任何外部 service
- 仅 stdlib + 静态 binary，可单独运行

**Docker image 状态**：
- `Dockerfile` 存在（multi-stage `rust:1.82` + `golang:1.22` → `debian:bookworm-slim`）
- `deploy/console-stack.yml` 包含 `contextforge` service + healthcheck + persistent volume
- ⚠️ **当前 image tag 是 `contextforge-daemon:v0.3.0`** (`deploy/console-stack.yml:54`) — 是本地 `build:` 段产物，**未推送公开 registry**。Console 团队接入 2 选 1：
  - (a) 自行 `docker compose -f deploy/console-stack.yml build contextforge` → tag own registry
  - (b) 等 ContextForge 团队 ship `ghcr.io/<org>/contextforge:v0.7.0` (follow-up；可在 cross-repo notify 时反馈优先级)

**Binary release artifacts**：
- ⚠️ **当前无 GitHub Release tarball / 平台二进制公开下载**（README §"v0.2 limitations" 仍标注 "No GitHub Release tarball is published from this repo yet"）
- v0.7.0 ship from source per README §Quick Start：clone repo → `cargo build -p contextforge-core --release && go build -o contextforge ./cmd/contextforge`

### 3. 认证模式

**完整实现** `internal/consoleapi/router.go:68-81` + `internal/cli/console_api_serve.go:47,67-69`：

| 模式 | 触发条件 | 行为 |
|---|---|---|
| **trusted-network** (默认) | `--auth-token` 空 + env `CONTEXTFORGE_CONSOLEAPI_AUTH_TOKEN` 空 | 直接 inner.ServeHTTP，**不检查 Authorization header**。CLI startup log 印 `auth=trusted-network` |
| **bearer-token** | `--auth-token <X>` 或 env `CONTEXTFORGE_CONSOLEAPI_AUTH_TOKEN=<X>` | 每请求需 `Authorization: Bearer <X>`，`crypto/subtle.ConstantTimeCompare` 比对，不匹配返 **401 + `{"error":{"code":"UNAUTHORIZED","message":"missing or invalid bearer token"}}`** |

**与 Console 期望对齐性**：
- Console `CONSOLE_API_CORE_AUTH_MODE=trusted-network` 默认 → ContextForge default trusted-network → ✅ **完全一致**
- Console docker-compose 已配 `CONSOLE_API_CORE_AUTH_MODE: trusted-network` (`deploy/console-stack.yml:80`) → 无需注入 token

**Token 来源优先级**：
1. `--auth-token` CLI flag（最高）
2. env `CONTEXTFORGE_CONSOLEAPI_AUTH_TOKEN`（CLI flag 默认值）
3. 都空 → trusted-network（无 auth）

**mTLS**：⚠️ **v0.7 未实现**。当前所有 traffic 都是 plain HTTP，依赖 trusted-network deployment（loopback / private subnet / docker network）保护。如 Console 部署在跨主机 / 公网，需自行通过 reverse proxy (nginx/Caddy/Envoy) 加 TLS termination。mTLS 原生支持留 v1.x（无 ADR proposal yet）。

### 4. X-Confirm header 完整规则

**实现**：`internal/consoleapi/router.go:54-63` `confirmMiddleware`

**适用 endpoint（共 3 个，**仅**这 3 个，其它 destructive op 不需要**）**：

| # | Endpoint | 适用 |
|---|---|---|
| W4 | `PATCH /v1/workspaces/{id}/config` | ✅ 需 X-Confirm |
| M4 | `POST /v1/memory/{id}/deprecate` | ✅ 需 X-Confirm |
| M5 | `POST /v1/memory/{id}/soft-delete` | ✅ 需 X-Confirm |
| **J3 cancel** | `POST /v1/index-jobs/{id}/cancel` | ❌ **不需要** X-Confirm（index-job cancel 是 co-operative，server 端无破坏性） |
| 其它 destructive (e.g. workspace delete) | — | v0.7 未提供 |

**Value 与 OR 语义**：
- header 形式：`X-Confirm: yes`（**value 必须固定字面量 `"yes"`**；`X-Confirm: true` / `X-Confirm: 1` 都不通过）
- query 形式：`?confirm=true`（**value 必须固定字面量 `"true"`**）
- **OR 语义**：header **OR** query，**任一满足即通过**（ADR-017 D2）

**不带任一时**：
- Status: **412 Precondition Failed**
- Body shape:
  ```json
  {"error":{"code":"PRECONDITION_FAILED","message":"X-Confirm: yes header or ?confirm=true query required for destructive op (ADR-017 D2)"}}
  ```

**Console BFF 接入建议**：HTTPAdapter 在调 `Deprecate` / `SoftDelete` / `PatchConfig` 时自动注入
`X-Confirm: yes` header；ops curl / 手工 debugging 用 `?confirm=true` 更易在 shell history 看到痕迹。

### 6. Error response shape

**统一 ErrorBody**（`internal/consoleapi/types.go:132-137` + `router.go:90-98`）：

```json
{
  "error": {
    "code":    "NOT_FOUND",
    "message": "memory item not found: mem-xxx"
  }
}
```

**Code 枚举与 status 映射**（`router.go:106-121`）：

| HTTP Status | Code | 触发条件 |
|---|---|---|
| 400 | `BAD_REQUEST` | malformed JSON / 必填字段缺失 / missing path id / 不支持的 query 参数（如 `?status=cancelled`） |
| 401 | `UNAUTHORIZED` | bearer-token mode 下 token 缺失或不匹配 |
| 404 | `NOT_FOUND` | workspace/job/chunk/trace/memory/eval id 不存在 |
| 409 | `CONFLICT` | cancel 已 terminal 的 job (`ErrJobTerminal`) |
| 412 | `PRECONDITION_FAILED` | X-Confirm 缺失（PATCH config / memory deprecate / memory soft-delete） |
| 503 | `SERVICE_UNAVAILABLE` | 数据面 unreachable（ADR-016 D4；触发 Console 切 mock / 退化 UI） |
| 500 | `INTERNAL` | 其它兜底（未分类异常） |

**Console HTTPAdapter 接入建议**：
- 4xx body 透传 + sentinel error 映射（已实现）
- 5xx 包到 `ErrCoreUnavailable` 触发 retry + degraded UI banner
- **412 不要计入 retry**（precondition 是 client error，retry 也会失败；应触发"补 X-Confirm 后重试"前端逻辑）

**示例 body**（按 status）：

```json
// 404
{"error":{"code":"NOT_FOUND","message":"workspace not found: ws-foo"}}
// 409
{"error":{"code":"CONFLICT","message":"job already terminal: succeeded"}}
// 412
{"error":{"code":"PRECONDITION_FAILED","message":"X-Confirm: yes header or ?confirm=true query required for destructive op (ADR-017 D2)"}}
// 503
{"error":{"code":"SERVICE_UNAVAILABLE","message":"data plane unavailable"}}
```

### 7. Observability events 数据契约

**handler**: `internal/consoleapi/handlers.go:481-497`

**协议形态**：**batch long-poll polling**，**不是 SSE**。
- Content-Type: `application/json`
- Body: JSON array `[]ObservabilityEvent`（即使空也返 200 + `[]`，**不返 204**）

**Query params**：
| param | 默认 | clamp | 含义 |
|---|---|---|---|
| `wait` | `30s` | `[1s, 60s]` | 设计意图为 long-poll 阻塞时长 |
| `limit` | `100` | `[1, 500]` | 单次 batch 最多 event 数 |

⚠️ **重要 caveat — v0.7 `?wait=` 当前未在 REST tier 真正阻塞**：
- `handlers.go:485` 显式 `_ = parseWaitParam(r)`（结果丢弃，未传给 `deps.Events.Recent`）
- 当前行为：handler 立即调 `deps.Events.Recent(limit)` 返回 cached 最近 N 条，**不 block 等新事件**
- README v0.4 描述的 "long-poll" 是设计意图；REST 层 long-poll 真实实现 [SPEC-DEFER:task-future.consoleapi-sse-or-long-poll-honoring]
- gRPC tier (`EventsService.StreamEvents`) 是真 streaming，Go REST 包一层后 degrade 到 batch

**对 Console 影响**：
- ✅ `GET /v1/observability/events` 可以**轮询调用**，每次返回最近 N 个事件
- ⚠️ **不要假设 `wait=30s` 真的会 block 30 秒**；轮询频率应由 Console 端控制（如每 5s 一次）
- ⚠️ 高频率轮询 + 同 cache → **会重复读到旧事件**

**`?since=<timestamp>` cursor**：⚠️ **v0.7 未实现**。当前接口只返"最近 N 条"，无增量游标。后续 v1.x 加 `?since=<rfc3339>` 留 [SPEC-DEFER:task-future.events-since-cursor]。

**ObservabilityEvent 完整字段**（`internal/contractv1/contractv1.go:262-272`）：
```go
type ObservabilityEvent struct {
    EventID      string             `json:"event_id"`       // 形如 "evt-<nanos>"
    EventType    string             `json:"event_type"`     // 见枚举
    Severity     string             `json:"severity"`       // info / warn / error
    Source       string             `json:"source"`         // "consoleapi" / "indexjob" / "search" / ...
    Message      string             `json:"message"`        // human-readable
    Timestamp    time.Time          `json:"timestamp"`      // RFC3339Nano UTC
    TraceID      *string            `json:"trace_id"`       // nullable
    JobID        *string            `json:"job_id"`         // nullable; e.g. "job-7" for indexjob events
    Availability FieldAvailability  `json:"field_availability"`
}
```

**EventType 已知枚举**（v0.7 实际发送）：
- `workspace.created` / `workspace.updated`
- `indexjob.enqueued` / `indexjob.cancelled`
- `indexing.progress` / `indexing.cancelled` / `indexing.error`（Rust JobRunner heartbeat → EventBus）
- (后续 Phase 加 audit-driven 事件如 `memory.pin` / `memory.deprecate` / `memory.soft_delete` 可能从 Rust audit sink 透传，但 v0.7 暂未跨 audit→eventbus 桥接，需在 Console UI 自己 deduce from memory updated_at)

**Ring cap**: 1000（`memstore.go:53`）；超出最老 evict。

**SSE 升级 / Event streaming 真实现** [SPEC-DEFER:task-future.consoleapi-sse]，留 v1.x。

## P2 — 加分项

### 9. Health endpoint 详细结构

⚠️ **重要 caveat — v0.7 `/v1/health` 是 binary healthy/degraded，不暴露 5 链路细分**

**当前 response shape**（`handlers.go:21-57`）：
```json
{
  "status":           "healthy",        // "healthy" | "degraded"
  "contract_version": "v1",             // 常量
  "last_connected_at": "2026-05-24T...Z", // nullable
  "error_reason":      null,            // string; degraded 模式带 reason 文案
  "missing_must_have_fields": []        // FieldAvailability array; 列缺失的 must-have 字段
}
```

**3 种 backend mode 输出**：

| BackendKind | HTTP | status | missing_must_have_fields |
|---|---|---|---|
| `grpc` (默认，gRPC 正常) | **200** | `healthy` | `[]` |
| `inmem-fallback` (env `CONSOLE_API_FALLBACK_INMEM=1`) | **200** | `degraded` | `[{"object":"core","missing_must_have_fields":["data_plane_persistence"]}]` |
| `degraded` (gRPC 不可达 + inmem 未启用) | **503** | `degraded` | `[{"object":"core","missing_must_have_fields":["data_plane"]}]` |

**Console UI CoreHealthCard 5 链路（db / index / embed / retriever / eval）期望**：
- ⚠️ **当前 ContextForge 不暴露分链路状态**
- Console UI 2 选 1：
  - (a) 用 `MissingMustHaveFields` 数组反推（粗粒度，只能区分 healthy vs degraded vs unreachable）
  - (b) 写死 5 链路 UI；ContextForge 后续 v0.7.x 或 v0.8 ship `/v1/health?detailed=true` 加细分（cross-repo 反馈优先级）
- 推荐 (b) + 在 Console UI 加 "v0.7 only reports overall status; per-component breakdown coming in v0.8" 注释

### 10. Long-running operation 行为

**Index-job**：
- 无 hardcoded server-side timeout（依赖 client / OS keep-alive）
- 文件大小上限：5MB / file（task-3.x 锁定；超出 skip + 计入 `skipped_files`）
- 文件数无 hardcoded 上限；大 repo 实测可达数千文件
- 典型完成时间（fixture `test/fixtures/index-job-real/`，含 ~3 个 md）：< 5s
- Console 端 GET 轮询频率建议：1s 一次直到 `status ∈ {succeeded, failed, cancelled}`

**Eval-run**：
- **5min hardcoded ctx timeout**（`internal/consoleapi/eval_runner.go:30`）；超时 status=failed + error_message=ctx.Err()
- 当前 light-weight harness（BuiltinGoldenQuestions mock pass-all）→ < 100ms 即 terminal
- v1.x retriever-backed recall 启用后预期 ~30s-2min（dataset 大小相关）
- Future `?timeout=<duration>` query 参数留 [SPEC-DEFER:phase-15.eval-cancel + phase-future.eval-runtime-config]

**Cancel 行为**：
| 场景 | Status | Body |
|---|---|---|
| Cancel queued/running job | **204** | （空） |
| Cancel terminal (succeeded/failed/cancelled) job | **409** | `{"error":{"code":"CONFLICT","message":"job already terminal: <status>"}}` |
| Cancel 不存在 id | **404** | `{"error":{"code":"NOT_FOUND","message":"index job not found: ..."}}` |
| Cancel cooldown | **无** | 同一 id 重复 cancel 第二次直接命中 409 |
| Eval cancel | **不支持 v0.7**（无 cancel endpoint） | [SPEC-DEFER:console-eval-cancel]（Console contract 22 endpoint 本身不含 eval cancel） |

### 11. Logs / metrics endpoint

⚠️ **v0.7 不提供 `/v1/metrics` 或 `/v1/logs` 端点**：
- 无 Prometheus exporter
- 无 unified logs HTTP endpoint
- 当前唯一 observability surface 是 `GET /v1/observability/events`（§7）

**未来留口**：
- Prometheus `/metrics` 留 v0.8 或 v1.x（无 ADR proposal yet）

**Console UI v1.x 观测面板建议**：
- 短期：基于 `/v1/observability/events` 轮询拉事件流 + 自行聚合
- 中期：cross-repo 反馈 metrics endpoint 需求优先级
