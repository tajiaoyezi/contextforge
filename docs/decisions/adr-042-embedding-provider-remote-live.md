## Context

ContextForge 截至 Phase 22（embedding-provider-completion, Done / v0.15.0）已把 embedding provider 推进到**完整抽象 + 工厂选型成立**：`core/src/embedding/factory.rs` 的 `select_provider`（`factory.rs:27`）把 config provider 名 + 请求 dim 映射到 `Arc<dyn EmbeddingProvider>`——`""` / `"deterministic"` → model-free `DeterministicEmbeddingProvider`（默认 byte-equiv Phase 19，ADR-027 D1）、`"fastembed"` → feature-gated `FastEmbedProvider`、`"remote"` → feature-gated `RemoteEmbeddingProvider`（`factory.rs:49`，从 env 读 `CONTEXTFORGE_REMOTE_ENDPOINT` / `_MODEL` / `_PROVIDER` / `_API_KEY`，api_key 永不记录）；dim 协商经 `negotiate_dim`（`factory.rs:88`，冲突 → `DimMismatch`，不静默截断 / 补零）。remote 路径的请求构造（`build_request_body`，`remote_provider.rs:60`）与响应解析（`parse_response`，`remote_provider.rs:74`）是**纯函数 + 契约层已实现**——契约测试（TEST-22.3.1~22.3.4）以 fixture 断言其正确、**从不触网**（ADR-013，CI 无密钥）。但**真实远程 embedding 端点的端到端 live 联调 + 实测语义召回从未真实跑过**——ADR-027 在 v0.15.0 ratify 时如实记 remote 真实联调 / 召回为 honest-defer `[SPEC-DEFER:phase-future.embedding-provider-remote]`，根因是 CI 无远程 API 密钥（且 remote 是付费外部 API、无免费 service container）。

**当前 in-repo embedding 的真实状态（本 phase 要关闭的诚实缺口）**：

- remote provider 仅有**契约层 + 纯函数构造**（`build_request_body` / `parse_response` fixture 契约测试，`remote_provider.rs:60` / `:74`），**不是**对真实远程端点的 live 召回测量。据 ADR-013，纯函数契约测试证明请求/响应序列化正确，但不能冒充真实语义召回背书。
- ADR-027 Ratification 如实写：remote 真实联调 / 召回 / 探针命中无远程密钥 → honest-defer，无召回数值，provider 层 phase。
- in-repo 唯一的「召回数」证据是 Phase 21 dogfood eval（deterministic / hybrid 路径）与 Phase 36 的 qdrant model-free KNN recall——**均非真实 embedding model 的语义召回**。

**DE-RISK 已由 lead 真实证实（不是推测——真实端到端 round-trip 已跑通）**：SiliconFlow（`https://api.siliconflow.cn/v1/embeddings`，OpenAI-compatible）+ 模型 `Qwen/Qwen3-Embedding-8B` 端到端 round-trip 真实跑通——native dim=4096，OpenAI 风格 `dimensions` 参数被接受并生效（Matryoshka，本 phase 用 1024），CJK 输入正常；Rust `RemoteEmbeddingProvider` → `ureq` → `parse_response` 路径在 Windows MSVC 下 `--features embedding-remote` 真实编译并跑通、返回 1024 维向量。这证明 remote 读写路径 + OpenAI-compatible 语义在真实付费端点上正确，本 phase 的剩余工作是把它系统化为可复现的语义 recall harness + config env-bridge，而非验证可行性本身。

本 Phase 37 是承 Phase 22（embedding provider 抽象）血脉的**深化与真实兑现**——把 ADR-027 honest-deferred 的 remote provider live 维度**真实兑现**：(1) 建一个 env-gated live 语义 recall harness，在作者手工标注集上测 real remote 模型 vs deterministic（model-free）基线的 recall@1 / recall@3；(2) 对一个 REAL 远程 OpenAI-compatible 端点跑（de-risk 已证）；(3) 把 Go `[remote]` config 段经 env-bridge 接进 spawned core daemon，兑现 `factory.rs:52` 注释里的 config plumbing follow-up。全程守 ADR-013：live 语义召回真实数字一律**真实跑出后据实记录**（task-37.1 §10 引用主 agent 本机真实 SiliconFlow run），floor 是 guard、real-vs-baseline delta 是报告，**小型手工标注集不夸大为大基准质量断言**；tag/digest/tlog 标 `待回填` 直至发版后回填（ADR-013 不预填）。默认构建不变：`embedding-remote` 仍 opt-in/off，默认构建 0 网络 / 0 远程依赖（ADR-004/008）；**0 新依赖**（`ureq` 自 task-22.3 起已是 `embedding-remote` feature 下的 optional dep）。**API KEY 永不进 config.toml**——仅由用户设 env（PRD 安全基线）。

## Decision

remote embedding live 召回采用 **「env-gated live 语义 recall harness（real remote 模型 vs deterministic 基线对照 + 作者手工标注集 + 无密钥诚实退出）+ 真实实测召回数（real-vs-baseline delta，据本机真实 run 据实记录）+ remote-embedding-config-bridge（Go `[remote]` Model add-only + setRemoteEnv env-bridge，env-wins，API key env-only）+ 默认 0-network / 0 新 dep / 既有契约不变」** 策略，分 4 个决策点：

### D1 — live 语义 recall harness 方法论（real remote 模型 vs deterministic 基线对照；作者手工标注集；env-gated 无密钥诚实退出）（task-37.1）

新增 `core/tests/remote_embedding_recall.rs`，`#![cfg(feature = "embedding-remote")]`，env-gated on `CONTEXTFORGE_REMOTE_API_KEY`（factory 另读 `CONTEXTFORGE_REMOTE_ENDPOINT` / `_MODEL` / `_PROVIDER`；api_key 永不记录）。当 `CONTEXTFORGE_REMOTE_API_KEY` 未设 → `eprintln` 一条 skip notice + `return`（honest-defer——CI 无密钥时干净跳过、**不** fail，ADR-013）。harness 方法论：

- **作者手工标注语义集**：作者手工标注 15 个 case / 16 个文档，覆盖 英文复述 / 代码概念 / CJK / 跨语言；且故意埋近义干扰项（`config_save` vs `config_load`、`bm25` vs `hybrid`、`cjk_index` vs `cjk_vector`），使 top-k 命中**非词面 trivial**——real 模型须依赖语义（非共享词面）才能把标注相关文档排进 top-k。
- **同一标注集 + 同一精确路径双 provider 对照**：同一标注集 + 同一 `BruteForceVectorBackend` 精确余弦路径上，比较 real remote 模型 vs deterministic（model-free）基线。`VectorChunk { chunk_id: ChunkId(String), embedding: Vec<f32>, metadata: Option<serde_json::Value> }`；`VectorIndexConfig { dim: 1024, metric: VectorMetric::Cosine, persistence_path: None, collection_id: String }`。
- **recall 测量**：对每个 case 算 BruteForce 精确 top-3，判 `recall@1`（标注相关文档排第 1）/ `recall@3`（标注相关文档在 top-3）；`recall@k = hit_k / n`。先 `eprintln` 出 **ACTUAL 实测数**（含每 case top3 + HIT@1/hit@3/MISS）再 `assert`：floor `recall@3 >= 0.70`（防回退 guard）且 `remote recall@1 > deterministic recall@1`（real embeddings 须跑赢 model-free 基线，否则真实模型买不到东西）。floor 是 guard、real-vs-baseline delta 是报告（真实值据本机真实 run 据实记录，ADR-013 不伪造）。
- **非网络 well-formed 守护测试**：另有一个非网络 well-formed 守护测试（doc id 唯一 / 每个 relevant id 存在 / case 数 >= 12），**无密钥也总是跑**——使 harness 在无密钥环境也有确定性逻辑校验。

**0 新依赖、0 schema migration、0 proto 改动、0 默认构建变更**。Tests：TEST-37.1.1（live 语义 recall harness，env-gated——对 live remote 端点实测 `recall@3 >= floor` 且 `remote recall@1 > deterministic recall@1`）+ TEST-37.1.2（标注集 well-formed 守护——**无密钥** 也跑，断言 doc id 唯一 / relevant id 存在 / case 数 >= 12）+ TEST-37.1.3（= LAST，D2 lint）。

**理由**：real-vs-deterministic 基线对照是干净的语义召回指标——deterministic provider 无语义（hash 派生向量、排序近随机），real 模型应把标注相关文档排进 top-k，两者 delta 即真实 embeddings 实际买到的语义质量。作者手工标注集故意埋近义干扰（`config_save` vs `config_load` 等），使命中非词面 trivial；env-gated honest-defer（无密钥 → 干净 `return`，不 fail）使 harness 在无密钥环境（CI / 未配密钥的本机）干净跳过、不伪造召回通过。floor 作 guard（防回退）、real-vs-baseline delta 作报告（据实记录真实语义质量）——两者职责分离，呼应 ADR-006 recall-eval-acceptance-gate 的 gate 方法论与 ADR-041 的 floor-guard + 真实数回填先例。**诚实范围**：小型手工标注集，recall@3 floor 是回退 guard 而非质量上限断言；大语料语义质量续 `[SPEC-DEFER:phase-future.embedding-large-corpus-recall]`。

### D2 — 真实实测召回数（real-vs-baseline delta，据主 agent 本机真实 SiliconFlow run 据实记录；CI honest-defer 因 remote 是付费外部 API 无免费 service container）（task-37.1 §10）

harness（D1）跑出的真实语义 recall 数字是本 phase 的真实召回证据。据 ADR-013，主 agent 本机对真实 SiliconFlow 端点（`Qwen/Qwen3-Embedding-8B`，dim=1024）真实跑出：**real Qwen3 recall@1=0.8667(13/15) recall@3=1.0000(15/15)；deterministic recall@1=0.0000 recall@3=0.0667；delta @1=+0.8667 @3=+0.9333**。两个 @1 miss 恰是故意埋的硬近义干扰（`config_save` 挤掉 `config_load`、`hybrid` 挤掉 `bm25`）——即 real 模型把明显语义对排进 top-3（recall@3=1.0），仅在两个故意设计的硬近义对上 top-1 让位给干扰项。这些数字记入 task-37.1 §10 + ADR-042 §Ratification + `docs/releases/v0.30.0-evidence.md`，并标注是主 agent 本机真实 SiliconFlow run（非合成、非 CI）。

**CI honest-defer（与 qdrant 的诚实差异）**：与 Phase 36 qdrant 不同——qdrant 有免费 OSS service container 可接进 CI 拓扑（ADR-041 D3 永久守护）；remote embedding 是**付费外部 API、无免费 service container**，故 CI 无密钥时 harness 干净 honest-defer skip，召回由**本机已认证 run 实测**而非每次 CI run 守护。这是 remote 与 qdrant 的结构性差异，据 ADR-013 据实记录、不夸大为「CI 每次 run 守护」。

**理由**：ADR-013 禁伪造红线——召回数必须真实跑出后据实记录，不伪造合成数。本 phase 的 de-risk（real SiliconFlow + ureq round-trip 已证 1024 维向量返回正确）证明可行性，且主 agent 本机真实跑出了具体语义 recall 数（real recall@3=1.0000 vs deterministic recall@3=0.0667，delta @3=+0.9333）——这是真实 evidence，floor（recall@3 >= 0.70）是 guard、real-vs-baseline delta 是报告。**诚实判读**：小型手工标注集，recall@3=1.0 证明模型把明显语义对排在近义干扰之上，**非大基准质量断言**；大语料语义质量续 `[SPEC-DEFER:phase-future.embedding-large-corpus-recall]`。

### D3 — remote-embedding-config-bridge（Go `[remote]` Model add-only + setRemoteEnv env-bridge；env-wins；API key env-only 永不进 config；Rust 0-dep）（task-37.2）

兑现 `core/src/embedding/factory.rs:52` 注释里的 config plumbing follow-up：

- **Go `[remote]` Model 字段 add-only**：`internal/config/config.go` 的 `RemoteProviderConfig`（`config.go:52`）现有 `Enabled` / `Provider` / `Endpoint`，add-only 补 `Model` 字段（toml round-trip 保真，`assignRemote`（`config.go:321`）+ toml serialize（`config.go:204`）同步）。
- **新 `setRemoteEnv` helper**：镜像 Phase 34 的 `setVectorEnv`（`cmd/contextforge/main.go:288`）/ `setDataDirEnv`（`main.go:265`）跨进程 env-bridge——当 `[remote]` 段存在且对应 env 未设时，导出 `CONTEXTFORGE_REMOTE_ENDPOINT` / `_MODEL` / `_PROVIDER` 给 spawned core daemon；spawned core 经既有 `factory.rs:54-58` env 路径接收。**ENV WINS**（显式 env 覆盖 config 文件，向后兼容）；无 `[remote]` 段 → 不导出 → 默认行为不变。`setRemoteEnv` 接线进 doServe + doMCP（镜像 setVectorEnv 接线）。
- **安全红线**：API KEY 永不进 `config.toml`——`setRemoteEnv` 不处理密钥；`CONTEXTFORGE_REMOTE_API_KEY` 仅由用户设 env（PRD 安全基线）。
- **Rust core 0 toml dep**：复用既有 `CONTEXTFORGE_REMOTE_*` env 读取（`factory.rs:54-59`），0 改动 factory。

**0 新依赖、0 schema migration、0 proto 改动、0 默认构建变更**。Tests：TEST-37.2.1（Go config round-trip——`[remote]` Model 字段 toml 写读保真）+ TEST-37.2.2（`setRemoteEnv` env-bridge——`[remote]` 段存在 + env 未设 → 导出三变量；env 已设 → env-wins 不覆盖；无 `[remote]` 段 → 不导出）+ TEST-37.2.3（= LAST，D2 lint）。

**理由**：`factory.rs:52` 注释明记 config plumbing 是 follow-up——remote provider 现仅经 env 配置，Go config 文件的 `[remote]` 段无法把 endpoint/model/provider 传给 spawned core。`setRemoteEnv` 复用 Phase 34 已验证的 `setVectorEnv` / `setDataDirEnv` 跨进程 env-bridge 模式（best-effort `config.Load` → 逐变量 `os.LookupEnv` env-wins 守卫 + `os.Setenv` + restore 闭包），把 `[remote]` config 接进 spawned core 的既有 env 路径——既兑现 config plumbing、又守 env-wins 向后兼容 + API key env-only 安全红线。Rust core 0 toml dep（复用既有 env 读取），符合 ADR-004 local-first（Go 侧承担 config 文件解析、Rust 侧只读 env）。

### D4 — 默认 0-network baseline + embedding-remote 仍 optional/off + 既有契约不变 + 0 新依赖（all tasks）

所有改动保持默认构建 0 网络 + 0 远程依赖 + 既有行为 / proto / 既有契约不变（ADR-004）+ 0 新依赖（ADR-008）：

- `embedding-remote` 仍 opt-in/off（`core/Cargo.toml` feature gate）：`remote_embedding_recall.rs` `#![cfg(feature = "embedding-remote")]`，默认 `cargo test --workspace`（无 remote feature）**不编译它**，默认 embedding 路径仍经 model-free `DeterministicEmbeddingProvider`（ADR-027 D1，byte-equiv Phase 19）。
- **0 新依赖**：`ureq` 自 task-22.3 起已是 `embedding-remote` feature 下的 optional dep（`core/Cargo.toml`）；本 phase 不引入任何新 direct dep，`Cargo.lock` 默认构建段不变。Go 侧 add-only `Model` 字段 + `setRemoteEnv` 复用既有 `internal/config` + stdlib（0 新 Go dep）。
- Go `[remote]` Model 字段 + `setRemoteEnv` 是 add-only（既有 `Enabled` / `Provider` / `Endpoint` + `setVectorEnv` / `setDataDirEnv` 接线不变）；默认构建 + 既有行为 + proto + migration 零变更。
- 既有 `cargo-test` / `go-test` / `lint` / `spec-lint` / `feature-build` 各门不退化；`embedding-remote` feature-build 是既有 feature-build 矩阵的一员（仅 build，不在 CI 跑 live harness——无密钥 honest-defer）。

**理由**：ADR-004 local-first + ADR-008 dep add-only——默认构建 0 网络 / 0 远程依赖 / 0 新依赖是不可让渡 baseline。feature-gated remote 是「按需启用的能力」而非「默认引入的成本」，与 D1-D3 的 live 召回兑现正交：harness 在 `embedding-remote` feature 边界内，setRemoteEnv 仅在 `[remote]` 段存在时导出 env，默认 `cargo test --workspace` 与 Phase 19 语义 baseline 字节等价。API key env-only（永不进 config.toml）守 PRD 安全基线。

## Consequences

- **Positive**: remote embedding 首次真实端到端联调 + 实测语义召回（D1 env-gated harness：real remote 模型 vs deterministic 基线对照 over 作者手工标注集，floor-guard + real-vs-baseline delta 报告，TEST-37.1.1/37.1.2，0 dep / 0 proto / 0 migration）；真实语义召回数据据主 agent 本机真实 SiliconFlow run 据实记录（D2 real recall@3=1.0000 vs deterministic recall@3=0.0667 / delta @3=+0.9333，ADR-013 真实非伪造、小集不夸大为大基准）；Go `[remote]` config 经 setRemoteEnv env-bridge 接进 spawned core 兑现 `factory.rs:52` plumbing follow-up（D3 Model add-only + env-wins + API key env-only，TEST-37.2.1/37.2.2）；全部默认构建 0 网络 / 0 远程依赖 / 0 新依赖（D4 `embedding-remote` opt-in/off + `ureq` 早已 optional，ADR-004 / ADR-008），既有各门不退化。**直接兑现 ADR-027 的 remote provider 维度**：其 v0.15.0 时 remote 真实联调 / 召回的 honest-defer now 推进为 real live 联调 + 语义 recall measured（以 add-only Phase-37 Amendment 记 ADR-027 embedding-provider-remote fulfilled，不溯改其 D-body，ADR-014 D5）。
- **Negative / open**（受阻 / 延后项如实，不伪造、不夸大）：CI honest-defer——remote 是付费外部 API、无免费 service container，CI 无密钥时 harness 干净 honest-defer skip，召回由本机已认证 run 实测（**与 qdrant 不同**——qdrant 有免费 OSS service container 可 CI 每次 run 守护，remote 不可）`[SPEC-DEFER:phase-future.embedding-remote-ci-credential]`；大语料语义召回质量（标准化大基准）→ honest-defer `[SPEC-DEFER:phase-future.embedding-large-corpus-recall]`——本 phase 是小型手工标注集证明模型把明显语义对排在近义干扰之上、非大基准质量断言；多 remote provider 端点联调（Cohere / 其它 OpenAI-compatible）→ honest-defer `[SPEC-DEFER:phase-future.embedding-multi-provider-live]`；remote reranker（cross-encoder over HTTP）端点联调 → honest-defer `[SPEC-DEFER:phase-future.embedding-remote-reranker-live]`；远程探针命中 / 健康度 live 守护 → honest-defer `[SPEC-DEFER:phase-future.embedding-remote-health-probe]`；env-gated harness 在无密钥环境（CI / 未配密钥的本机）干净 skip（无 key → `eprintln` + `return`，不 fail）——这是 by-design honest-defer 不是缺口——以上据 ADR-013 如实分级、不伪造完成、不夸大缺口。
- **Ratification**: 本 ADR **Proposed**。task-37.1（harness + 本机真实 run）/ 37.2（config-bridge）通过后于 v0.30.0 closeout（task-37.3）据真实 run / 实测 recall 逐 D ratify Proposed→Accepted（见 §Ratification）；真实 recall 数据据主 agent 本机真实 SiliconFlow run 据实记录，tag/digest/tlog 待回填，绝不预填（ADR-013）。
- **Follow-ups**: CI 凭据 / 付费 API live 守护 `[SPEC-DEFER:phase-future.embedding-remote-ci-credential]`；大语料语义召回质量 `[SPEC-DEFER:phase-future.embedding-large-corpus-recall]`；多 remote provider 端点联调 `[SPEC-DEFER:phase-future.embedding-multi-provider-live]`；remote reranker 端点联调 `[SPEC-DEFER:phase-future.embedding-remote-reranker-live]`；远程探针命中 / 健康度 live 守护 `[SPEC-DEFER:phase-future.embedding-remote-health-probe]`。ADR-027 的 `[SPEC-DEFER:phase-future.embedding-provider-remote]` 经本 phase D1/D2（live harness + 真实联调 + 实测召回）**兑现**，以 add-only Phase-37 Amendment 记其 fulfilled、不溯改 ADR-027 D-body 正文（ADR-014 D5，将在 task-37.3 closeout 追加）；ADR-006 / ADR-041 / ADR-004 / ADR-008 / ADR-013 引用均不溯改其正文。

## Alternatives

- **A1（纯函数契约测试充当召回背书）**：用已有的 `build_request_body` / `parse_response` fixture 契约测试（`remote_provider.rs` tests）充当 remote 召回背书。否决：纯函数契约测试证明请求/响应序列化正确，**不是真实语义召回测量**（违 ADR-013 禁伪造红线）——它不触网、不验证真实模型的语义排序；据 D1/D2，真实召回须由 harness 对 live 远程端点真实跑出后据实记录，floor 是 guard、real-vs-baseline delta 是 evidence。
- **A2（合成向量 / 伪造远程响应充当 real 召回）**：用本地合成向量或伪造的远程响应充当 real embedding 召回。否决：合成 / 伪造向量**无真实语义**——deterministic 基线已是 model-free 对照项（其 recall@3=0.0667 恰证 model-free 无语义），用合成数冒充 real 模型召回会伪造语义质量（违 ADR-013）；据 D2，real 召回须由真实远程模型（SiliconFlow Qwen3）真实跑出，real-vs-deterministic delta 才是真实 evidence。
- **A3（remote 接 CI service-container 守护，镜像 qdrant）**：把 remote embedding 接进 CI service-container 使召回每次 CI run 守护（镜像 ADR-041 D3 qdrant）。延后 `[SPEC-DEFER:phase-future.embedding-remote-ci-credential]`：remote 是**付费外部 API、无免费 OSS service container**（与 qdrant 的 `qdrant/qdrant` 免费镜像不同）——无法像 qdrant 那样接进 CI 拓扑零成本守护；CI 无密钥时 harness 干净 honest-defer skip，召回由本机已认证 run 实测。这是 remote 与 qdrant 的结构性差异，据 ADR-013 据实分级、不强行套用 qdrant 的 CI 守护模式。
- **A4（大语料标准化基准）**：以标准化大语料基准（如 MTEB 子集）为 ground truth 测 recall。延后 `[SPEC-DEFER:phase-future.embedding-large-corpus-recall]`：作者手工标注集（15 case / 16 doc，故意埋近义干扰）是 model-dependent 语义召回的干净 primary——证明真实模型把明显语义对排在近义干扰之上（recall@3=1.0 vs deterministic 0.0667）；大语料标准化基准引入数据集体量 + 外部依赖，非本 phase scope，作 follow-up 延后而非本 phase primary。

## Ratification (v0.30.0 / task-37.3)

**Proposed → Accepted**（逐 D 据主 agent 本机真实 SiliconFlow run + 实测 recall，ADR-013 真实非伪造；tag/digest/tlog 待回填）：

- **D1 live 语义 recall harness 方法论 — ratified ✅**：`core/tests/remote_embedding_recall.rs`（`#![cfg(feature = "embedding-remote")]`，env-gated `CONTEXTFORGE_REMOTE_API_KEY`，factory 另读 `_ENDPOINT` / `_MODEL` / `_PROVIDER`）落地——作者手工标注集（15 case / 16 doc，故意埋近义干扰 `config_save` vs `config_load` / `bm25` vs `hybrid` / `cjk_index` vs `cjk_vector`）经同一 `BruteForceVectorBackend` 精确余弦路径对照 real remote vs deterministic 基线，`recall@1` / `recall@3`；无密钥 honest-defer 干净 skip 不 fail。TEST-37.1.1（live）+ TEST-37.1.2（well-formed 守护，无密钥也跑）本地双绿。
- **D2 真实实测召回数 — ratified ✅（真实数据已由主 agent 本机跑出）**：主 agent 本机对真实 SiliconFlow 端点（`Qwen/Qwen3-Embedding-8B`，dim=1024）实测 `REMOTE-EMBED semantic recall over 15 labeled cases (dim=1024) | deterministic: recall@1=0.0000 recall@3=0.0667 | remote: recall@1=0.8667 recall@3=1.0000 | delta@1=+0.8667 delta@3=+0.9333`（此为其一 run 的逐字输出）。**跨 3 次 run**：recall@1 在 **0.8667–0.9333** 间波动、recall@3=**1.0000 三次稳定**（remote 模型/服务非完全确定）；@1 miss（1–2 个，随 run）恰落在故意埋的硬近义干扰对（`config_save`↔`config_load`、`hybrid`↔`bm25`）的 top-1 让位上。**诚实判读（ADR-013）**：小型手工标注集，recall@3=1.0 证明模型把明显语义对排在近义干扰之上 → 真实 embeddings 的语义质量真实证明（real == 语义正确排序，取代纯函数契约 fixture），**非大基准质量断言**；大语料语义召回质量须标准化大基准 → honest-defer `[SPEC-DEFER:phase-future.embedding-large-corpus-recall]`，不夸大为「已压测大语料」。floor=0.70 为不退化 guard，真实 recall@3=1.0000 留足余量。**CI honest-defer**：remote 付费 API 无免费 service container（与 qdrant 据实区分）→ 召回由本机已认证 run 实测、非 CI 每次 run 守护。
