## Context

ContextForge 截至 Phase 38（embedding-remote-reranker-live, Done / v0.31.0）已把 hybrid 融合与 reranker 在**内核 / 生产数据面**落地，但两个检索信号在**对外 console-api REST 表面不可达 / 不可见**：

**当前 in-repo 检索信号通路的真实状态（本 phase 要关闭的诚实缺口）**：

- **hybrid 信号无法经对外 console-api 抵达 core**：`proto/contextforge/v1/search.proto` 的 `SearchRequest` 自 Phase 21（task-21.1）已有 `hybrid = 8`（:28）、`RetrievalResult` 有 `hybrid_score = 15`（:51），`core/src/server.rs` 的 hybrid 路径（:328-376）经 `req.hybrid` 触发 `search_hybrid`（RRF 融合）→ `retrieval_method="hybrid"` + `hybrid_score`（:369）。但**对外 console 数据面通路缺三环**：(1) `proto/contextforge/console_data_plane/v1/console_data_plane.proto` 的 `SearchRequest` 只到 `semantic = 7`（:151-162）、**无 `hybrid` 字段**，`SearchResultItem` 只到 `vector_score = 16`（:203）、**无 `hybrid_score` 字段**；(2) `core/src/data_plane/search.rs` 的 `query()`（:241）只有 `if req.semantic {..} else {BM25}`（:282-315）、**无 hybrid dispatch 分支**；(3) Go console-api（`contractv1.SearchRequest` 无 `Hybrid`、`handleSearch` 无 `?hybrid` 解析、`grpcclient.Search` 不转发）。即对外 `POST /v1/search` 用户**只能** `?semantic=true`（task-20.1），无法请求 hybrid 融合检索。这与 `?semantic` 不同——`?semantic` 自 Phase 20（ADR-024）已贯通对外。
- **rerank provenance 在对外 REST 的可见性从未被断言**：reranker 自 Phase 38（ADR-043 D3）已在生产数据面 opt-in 接线（`reranker_from_env` → `with_reranker`，含 `data_plane/search.rs` semantic 分支 :291-295），rerank `reason` 标记（如 `IDENTITY_RERANK_REASON`）经既有链路端到端流通（Rust `SearchResult.reason` → proto `reason=14` → `grpcclient.protoToSearchResult` `Reason: p.Reason`（:624）→ `contractv1.SearchResult.Reason` → REST JSON）。但**无 smoke / 测试断言它在对外 `POST /v1/search` 响应可见**——基础设施已就位、端到端可见性未被验证。
- **`hybrid_score` provenance 字段在 console_data_plane 缺 v1 parity**：console_data_plane `SearchResultItem` 有 `vector_score = 16`（task-32.3）却无 `hybrid_score`，而 `v1 RetrievalResult` 有 `hybrid_score = 15`——对外 hybrid 命中无独立融合分 provenance 字段。

**历史延后边界 + 一处与既定设计的冲突（本 phase 据实校正）**：`[SPEC-DEFER:phase-future.console-api-hybrid-forward]`（ADR-025 §Follow-ups / task-21.1 §下游 / task-21.3 / README:350 / `console_smoke.sh:49-50` / `smoke_syntax_test.go:705-706` 记录）与其姊妹 `[SPEC-DEFER:phase-future.console-api-rerank-forward]`（task-21.2 / task-21.3 记录）均设想 console-api 以 per-request query param（`?hybrid=true` / `?rerank=true`）转发，镜像 Phase 20 的 `?semantic` 范式。**但 Phase 38（ADR-043 D3）此后确立 reranker 为服务端 env 驱动（`CONTEXTFORGE_RERANKER_PROVIDER`）、明确不是 per-request 参数**（ADR-004 local-first + D3：reranker 为服务端 opt-in、不随每查询参数切换）。故 `?rerank=true` per-request 转发与既定设计**冲突**：若实现它，对外请求会覆盖服务端 reranker 选型，破坏 env 驱动模型。本 phase 据实裁断（ADR-013）——`console-api-hybrid-forward` 兑现（hybrid 确是 per-request 算法路由 flag，与 `?semantic` 同类，不冲突）；`console-api-rerank-forward` **重界定**——reranker 保持 env 驱动，改交付 rerank provenance 可见性，`?rerank=true` per-request 控制记为被 D3 取代（superseded）、不实现。

本 Phase 39 是承 Phase 20（console-api 语义贯通）+ Phase 21（hybrid 融合）+ Phase 38（reranker 数据面接线）血脉的**对外可达性贯通**——**0 backend 算法改动**：core hybrid 融合（`search_hybrid` + RRF）+ reranker 数据面 opt-in（`reranker_from_env`）均已存在，本 phase 复用全部既有范式，只加 (a) console_data_plane proto 两个 add-only 字段、(b) console 数据面 hybrid dispatch 分支、(c) Go console-api 三处转发、(d) smoke / 测试端到端断言。默认行为不变：`hybrid=false` → 既有 semantic / BM25 字节等价（ADR-004/ADR-015 add-only）；reranker 默认 unset 字节等价无 rerank（ADR-043 D3 不变）；**0 新依赖**（ADR-008）；proto add-only（既有字段号冻结，ADR-015 D1）；0 migration。全程守 ADR-013：rerank-forward 重界定据实记录、不夸大为 per-request 实现；`hybrid_score` 端到端携带、非由 score 推断；tag/digest/tlog 标 `待回填` 直至发版后回填。

## Decision

console-api 检索信号转发采用 **「console_data_plane proto add-only（`hybrid` 请求 flag + `hybrid_score` 结果 provenance，既有字段号冻结）+ console 数据面 hybrid dispatch（镜像 server.rs hybrid 路径 + 数据面 semantic 分支结构，复用 search_hybrid + reranker_from_env opt-in）+ Go console-api 转发（contractv1 add-only flag/field + handleSearch ?hybrid OR-merge + grpcclient 转发/映射）+ rerank provenance 可见性（reranker 保持 env 驱动、端到端断言 reason 可见、?rerank per-request superseded）+ 默认 hybrid=false / reranker unset 字节等价 / 0 新 dep / proto add-only 既有契约不变」** 策略，分 4 个决策点：

### D1 — console_data_plane proto add-only + 数据面 hybrid dispatch（既有字段号冻结、镜像 server.rs hybrid 路径、默认 hybrid=false 字节等价）（task-39.1）

`proto/contextforge/console_data_plane/v1/console_data_plane.proto` add-only：`SearchRequest` 加 `bool hybrid = 8`（紧随 `semantic = 7`，镜像 `v1/search.proto:28`——默认 `false` → semantic-only / BM25 向后兼容）；`SearchResultItem` 加 `float hybrid_score = 17`（紧随 `vector_score = 16`，镜像 `v1 RetrievalResult.hybrid_score=15`——RRF 融合分、0 表示 semantic/BM25-only 命中、parity with v1）。**既有字段号 1-7 / 1-16 全冻结不动**（ADR-015 D1 add-only，既有 client 兼容）。`buf generate` 重生 Go + Rust stub。`core/src/data_plane/search.rs` `query()` 的 `let hits = if req.semantic {..} else {BM25}`（:282-315）改为 `if req.hybrid {..} else if req.semantic {..} else {BM25}`；hybrid 分支镜像 `server.rs` hybrid 路径（:328-376）+ 数据面自身 semantic 分支结构——model-free `DeterministicEmbeddingProvider::default()` + 0-dep `BruteForceVectorBackend::new()`（与数据面 semantic 分支 :283-284 一致，**非** `server.rs` 的 env-factory，保 console 数据面内部一致性）+ `enumerate_chunks` + `index_chunks_semantic` + `search_hybrid(&req.query, top_k)` + 复用 `reranker_from_env()` opt-in（同 semantic 分支 :291-295）；结果映射（:339-368）加 `hybrid_score: if h.retrieval_method == "hybrid" { h.score } else { 0.0 }`（镜像 `vector_score` 条件 :359-363，task-32.3 范式）。

**0 新依赖、0 schema migration、0 默认构建变更**。Tests：TEST-39.1.1（数据面 hybrid dispatch——`req.hybrid=true` ⇒ `query()` 走 `search_hybrid`、命中 `retrieval_method="hybrid"` + `hybrid_score` 非零；`req.hybrid=false` + `req.semantic=true` ⇒ 走 semantic 分支字节等价；两者皆 false ⇒ BM25 字节等价）+ TEST-39.1.2（proto add-only + `hybrid_score` 填充——`SearchRequest.hybrid` / `SearchResultItem.hybrid_score` 字段号 = 8 / 17、既有字段号不动；hybrid 命中 `hybrid_score = score`、非 hybrid 命中 `hybrid_score = 0.0`）+ TEST-39.1.3（= LAST，D2 lint）。

**理由**：console_data_plane proto 与 `v1/search.proto` 是 ADR-015 确立的两条独立检索契约——`v1` 自 Phase 21 已有 hybrid，console_data_plane 缺 parity 是已识别 add-only 演进点。proto add-only 新字段（既有字段号冻结）守 ADR-015 D1，既有 client 兼容；数据面 hybrid 分支镜像 `server.rs` 已验证的 hybrid 路径 + 数据面自身 semantic 分支两个范式（最小新增）；hybrid 分支用 hardcoded `BruteForceVectorBackend`（而非 `server.rs` 的 env-factory）保 console 数据面内部与既有 semantic 分支一致（env-factory backend 在 console 数据面是既有 semantic 分支同款 asymmetry，续延后 `[SPEC-DEFER:phase-future.console-data-plane-vector-backend-factory]`，不在本 phase 扩面）；`hybrid_score` 填充镜像 `vector_score` 条件 provenance 范式（融合分端到端携带、非由 score 推断，ADR-013）；默认 `hybrid=false` ⇒ 走既有分支字节等价（ADR-004 向后兼容）。

### D2 — Go console-api hybrid 转发 + rerank provenance 可见性（镜像 ?semantic 范式、对外 POST /v1/search 贯通）（task-39.2）

`internal/contractv1/contractv1.go` add-only：`SearchRequest` 加 `Hybrid bool`（json tag `hybrid`，镜像 `Semantic` :125 的 add-only opt-in 范式）；`SearchResult` 加 `HybridScore float32`（json tag `hybrid_score`，镜像 `VectorScore` :153 的 cross-repo add-only provenance 范式，ADR-014 D4——ContextForge-Console contractv1 镜像同字段）。`internal/consoleapi/handlers.go` `handleSearch`（:443-462）加 `if r.URL.Query().Get("hybrid") == "true" { body.Hybrid = true }`（镜像 `?semantic` :452-454）。`internal/consoleapi/grpcclient/grpcclient.go` `Search`（:364-390）加 `Hybrid: req.Hybrid`（镜像 `Semantic: req.Semantic` :372）+ `protoToSearchResult`（:609-643）加 `HybridScore: p.HybridScore`（镜像 `VectorScore: p.VectorScore` :623）。对外 `POST /v1/search`（body `{"query":"...","hybrid":true}` 或 `?hybrid=true`）⇒ `handleSearch` OR-merge ⇒ `contractv1.SearchRequest.Hybrid=true` ⇒ `grpcclient` 转发 `pb.SearchRequest.Hybrid=true` ⇒ console 数据面 `query()` hybrid dispatch ⇒ 响应 `retrieval_method="hybrid"` + `hybrid_score` + （reranker env opt-in 时）`reason` rerank marker。

rerank provenance 可见性：reranker 由 `CONTEXTFORGE_RERANKER_PROVIDER` env 服务端 opt-in（ADR-043 D3 不变，**本 phase 不加 `?rerank` 参数**）；`reason` 经既有链路（`grpcclient.protoToSearchResult` `Reason: p.Reason` :624）端到端流通——本 task 加 Go 测试 + smoke（task-39.3 step）断言其在对外 `POST /v1/search` 响应可见（reranker env opt-in 时）。

**0 新依赖、0 proto 再改（消费 D1 字段）、0 schema migration**。Tests：TEST-39.2.1（`handleSearch` `?hybrid` OR-merge——`?hybrid=true` ⇒ `body.Hybrid=true`；body `{"hybrid":true}` ⇒ `Hybrid=true`；两者皆无 ⇒ `Hybrid=false`，镜像 `TestTask201_HandleSearchSemanticORMerge`）+ TEST-39.2.2（`grpcclient` 转发 + 映射——`Search` 转发 `req.Hybrid` 到 `pb.SearchRequest.Hybrid`；`protoToSearchResult` 映射 `p.HybridScore` 到 `out.HybridScore`、`p.Reason` 到 `out.Reason`（rerank provenance 承载））+ TEST-39.2.3（= LAST，D2 lint）。

**理由**：`?hybrid` 转发镜像 Phase 20（ADR-024）已验证的 `?semantic` 对外贯通范式（contractv1 add-only flag + `handleSearch` OR-merge + `grpcclient` 转发），结构一致、最小新增；`HybridScore` 字段镜像 `VectorScore` 的 cross-repo add-only provenance 范式（ADR-014 D4）；rerank provenance 可见性是「断言既有链路端到端可见」而非新建机制——`reason` 自 Phase 38 已流通，本 task 验证它在对外 REST 表面可见，关闭「reranker 接线了但对外看不到 provenance」的诚实缺口。

### D3 — rerank-forward 重界定（reranker 保持 env 驱动、非 per-request；?rerank=true superseded by ADR-043 D3；交付 provenance 可见性而非 per-request 控制）（task-39.2 / task-39.3）

历史 `[SPEC-DEFER:phase-future.console-api-rerank-forward]`（task-21.2 / task-21.3 记录）设想 console-api 以 `?rerank=true` per-request 参数转发 rerank。本 phase 据实重界定：

- **reranker 保持服务端 env 驱动（ADR-043 D3 不变）**：reranker 由 `CONTEXTFORGE_RERANKER_PROVIDER` env 在服务端 opt-in（unset / "" / "none" → 无 rerank 字节等价；"identity" → `IdentityReranker`；"remote" → feature-gated `RemoteRerankerProvider`），数据面三处生产路径（含 `data_plane/search.rs`）经 `reranker_from_env()` → `with_reranker` 接线。本 phase **不**加 `?rerank` per-request 参数。
- **`?rerank=true` per-request 控制 superseded（不实现）**：per-request rerank 转发与 env 驱动模型冲突（对外请求覆盖服务端 reranker 选型）；据 ADR-043 D3（reranker 为服务端 opt-in、非 per-query），`?rerank=true` per-request 控制记为**被 D3 取代（superseded）**、不实现——以 add-only Phase-39 Amendment 记于 ADR-043，据实声明该 defer 的 per-request 口径被取代、不伪装为「实现了 per-request rerank 转发」（ADR-013）。
- **改交付 rerank provenance 可见性**：rerank `reason` 标记在对外 `POST /v1/search` 响应可见（reranker env opt-in 时）经端到端 smoke（task-39.3 step，`CONTEXTFORGE_RERANKER_PROVIDER=identity` ⇒ 响应 `reason` 含 rerank marker）+ Go 测试断言——这是 `console-api-rerank-forward` 维度按 provenance-visibility 口径的真实兑现（用户经对外 REST 能看到结果被 rerank 过 + 服务端 reranker 配置生效）。

**理由**：ADR-013 据实分级——不实现一个与既定设计（ADR-043 D3 env 驱动）冲突的 per-request 控制项，也不伪装它已实现；据实记其被取代、改交付不冲突且有真实价值的 provenance 可见性。这与 Phase 35 的三处 grounding 诚实校正、Phase 36 的 defer 据实关闭同源——据实裁断优于强行套用历史 defer 口径。

### D4 — 默认 hybrid=false / reranker unset 字节等价 + 0 新依赖 + proto add-only 既有契约不变（all tasks）

所有改动保持默认行为 / proto / 既有契约不变（ADR-004/ADR-015）+ 0 新依赖（ADR-008）：

- **默认 `hybrid=false` 字节等价**：`req.hybrid` / `contractv1.SearchRequest.Hybrid` 默认 `false` ⇒ console 数据面走既有 `if req.semantic` / BM25 分支、`hybrid_score` 为 `0.0` ⇒ 检索结果与当前字节等价；既有 console_data_plane / REST client（不设 `hybrid`）行为不变。
- **reranker 默认 unset 字节等价无 rerank**（ADR-043 D3 不变）：`CONTEXTFORGE_RERANKER_PROVIDER` unset ⇒ `reranker_from_env` 返回 `None` ⇒ 不 `with_reranker` ⇒ 无 rerank provenance marker、字节等价。
- **proto add-only 既有字段号冻结**（ADR-015 D1）：`SearchRequest.hybrid=8` + `SearchResultItem.hybrid_score=17` 为新字段，既有字段号 1-7 / 1-16 全冻结不动，既有 client 兼容（不破 wire 契约）。
- **0 新依赖**：proto add-only 字段 + 数据面 dispatch 分支 + Go 转发均复用既有 `search_hybrid`（task-21.1）/ `reranker_from_env`（task-38.2）/ `with_reranker`（task-21.2）/ `BruteForceVectorBackend` / `DeterministicEmbeddingProvider` / 手写 Go 转发 + `buf` 既有工具链；无新 Rust / Go direct dep。
- console 数据面 hybrid 分支复用 model-free `DeterministicEmbeddingProvider` + 0-dep `BruteForceVectorBackend`（与数据面既有 semantic 分支一致，0 网络，ADR-004 local-first）。
- 既有 `cargo-test` / `go-test` / `lint` / `spec-lint` / `feature-build` 各门不退化。

**理由**：ADR-004 默认行为不变 + ADR-015 proto add-only + ADR-008 dep add-only——默认 `hybrid=false` / reranker unset 字节等价 + proto 既有字段号冻结 + 0 新依赖是不可让渡 baseline。hybrid 是「按需启用的对外检索路由」而非「默认引入的成本」，与 D1-D3 的对外贯通正交：默认不设 `hybrid` 与当前字节等价，proto 新字段不破既有 wire 契约。

## Consequences

- **Positive**: 对外 console-api 首次贯通 hybrid REST 转发（D1 console_data_plane proto add-only `hybrid=8` + `hybrid_score=17` 既有字段号冻结 + 数据面 hybrid dispatch 镜像 `server.rs` 路径，`buf generate` 重生 stub，TEST-39.1.1/39.1.2，0 dep / 0 migration；D2 Go console-api `contractv1` add-only `Hybrid` / `HybridScore` + `handleSearch` `?hybrid` OR-merge + `grpcclient` 转发/映射，对外 `POST /v1/search` body / query param 暴露 hybrid，TEST-39.2.1/39.2.2）；rerank provenance 在对外 REST 可见（D2/D3，reranker 保持 env 驱动、`reason` 端到端 smoke + 测试断言可见，关闭「reranker 接线了但对外看不到」缺口）；全部默认 hybrid=false / reranker unset 字节等价 + 0 新依赖 + proto add-only 既有契约不变（D4，ADR-004/ADR-015/ADR-008）。**直接兑现 ADR-025 的 `[SPEC-DEFER:phase-future.console-api-hybrid-forward]` + 据实重界定 ADR-043 follow-up 的 `console-api-rerank-forward`**：以 add-only Phase-39 Amendment 记 ADR-025 hybrid-forward fulfilled + ADR-043 rerank-forward 按 provenance-visibility 口径 fulfilled（per-request superseded），不溯改各自 D-body（ADR-014 D5）。
- **Negative / open**（受阻 / 重界定项如实，不伪造、不夸大）：`?rerank=true` per-request 控制——据 ADR-043 D3（reranker 服务端 env 驱动）记为 superseded、不实现（与既定设计冲突），改交付 provenance 可见性 `[SPEC-DEFER:phase-future.console-api-rerank-forward]`（口径重界定，非缺口）；console 数据面 hybrid 分支用 hardcoded `BruteForceVectorBackend`（而非 `server.rs` 的 env-factory backend）——与数据面既有 semantic 分支一致，env-factory backend 接入 console 数据面续延后 `[SPEC-DEFER:phase-future.console-data-plane-vector-backend-factory]`（既有 asymmetry，非本 phase 引入）；Console UI 侧 hybrid / rerank explain 面板 `[SPEC-OWNER:phase-future.console-semantic-explain]`（跨仓库 Console 领域）；大语料 hybrid / rerank 对外 REST 召回质量基准 `[SPEC-DEFER:phase-future.vector-large-corpus-perf]`（本 phase 为信号贯通 + provenance 可见性 wiring 断言，非大基准召回质量；hybrid 融合质量 ADR-025 已 ratify、rerank 质量 ADR-043 已 ratify，不重测）——以上据 ADR-013 如实分级、不伪造完成、不夸大缺口。
- **Ratification**: 本 ADR Proposed（Phase 39 / v0.32.0 规划稿）。task-39.1（proto + 数据面 dispatch）/ 39.2（Go 转发 + rerank 可见性）合入后于 v0.32.0 closeout（task-39.3）据真实验证逐 D ratify Proposed→Accepted（见 §Ratification 骨架）；tag/digest/tlog 待 post-tag-push 回填（ADR-013 不预填）。

## Alternatives

- **A1（仅加 Go console-api `?hybrid` 转发而不改 console_data_plane proto / 数据面）**：在 Go 侧加 `?hybrid` 解析但不加 proto 字段 / 数据面 dispatch。否决：`hybrid` flag **无法抵达 core**——console_data_plane.proto `SearchRequest` 无 `hybrid` 字段、`grpcclient` 无处转发、数据面 `query()` 无 hybrid 分支；Go 侧解析了也是 no-op（违 ADR-013 据实——转发链路不通则非真实贯通）。据 D1/D2，proto add-only + 数据面 dispatch + Go 转发须端到端，缺一环则 hybrid 信号断在中途。
- **A2（实现 `?rerank=true` per-request 转发，镜像 `?hybrid`）**：把 reranker 也做成 per-request query param。否决：与 ADR-043 D3（reranker 服务端 env 驱动、非 per-request）**冲突**——per-request 会让对外请求覆盖服务端 reranker 选型，破坏 env 驱动模型 + ADR-004 local-first（reranker 选型是服务端配置而非每查询切换）；据 D3，`?rerank=true` per-request 记为 superseded、不实现，改交付 provenance 可见性（reranker 配置生效经 `reason` marker 对外可见，不矛盾 D3）。
- **A3（console 数据面 hybrid 分支采用 `server.rs` 的 env-factory backend 选型）**：hybrid 分支用 `select_vector_backend(resolve_vector_backend())`（镜像 `server.rs:342-343`）而非 hardcoded `BruteForceVectorBackend`。延后 `[SPEC-DEFER:phase-future.console-data-plane-vector-backend-factory]`：console 数据面既有 semantic 分支（:283-284）已用 hardcoded `BruteForceVectorBackend`（task-32.1 的 env-factory 只接入 `server.rs`、未接 console 数据面）——hybrid 分支镜像数据面自身 semantic 分支保内部一致性，env-factory backend 接入 console 数据面是既有 asymmetry 的统一扩面，作 follow-up 延后而非本 phase scope（避免在 hybrid 引入与数据面 semantic 分支不一致的 backend 选型）。
- **A4（`hybrid_score` 由 `score` + `retrieval_method` 在 Go 侧推断而非 proto 独立字段）**：不加 proto `hybrid_score=17`，Go 侧据 `retrieval_method=="hybrid"` 把 `score` 当融合分。否决：违 v1 parity（`v1 RetrievalResult` 有独立 `hybrid_score=15`）+ ADR-013（融合分应端到端携带、非由 score 推断——`score` 经 reranker 重排后可能已非 RRF 原始融合分）；据 D1，`hybrid_score=17` add-only 独立字段使融合分 provenance 端到端忠实携带，与 `vector_score=16`（task-32.3）同范式。
