## Context

ContextForge 截至 Phase 32（vector-backend-config-plumbing-and-completeness, Done / v0.25.0）已把向量后端推进到配置可接线 + 工厂覆盖完整 + provenance 对齐阶段。本 Phase 33 是**第二轮治理债清扫**（镜像 Phase 31 / ADR-036），对若干在调研中被发现存在 **survey 高估**的治理项据实收敛。**关键诚实校正**：grounding 复核发现 survey 对多项的描述过度了——本 ADR 必须反映下方据实校正后的 HONEST scope，这正是 ADR-013 的价值所在（如实分级、不夸大、不伪造）。逐维度调研结论：

- **L2 SQLite embedding 缓存无上界（真实但 opt-in-path）**：`core/src/embedding/cache.rs` 的 L1 是 `BoundedCache` FIFO（Phase 31，:27-66，`DEFAULT_EMBEDDING_CACHE_CAP=50_000`，:23），但 L2 `sqlite_put`（:153-161，`INSERT OR REPLACE INTO embedding_cache` @ :155）**无 cap → 行只增不减**。`embedding_cache` 表经 `CREATE TABLE IF NOT EXISTS`（:110-120，列 `content_hash/provider/dim/vector`、PK `(content_hash,provider)`、隐式 rowid）创建，**非编号 migration**。**诚实 caveat**：`with_sqlite`（:105-126）**无任何生产调用点**（test-only，cache.rs:331/337），shipped daemon 用 memory-only L1 → 这是 **opt-in-path 的纵深防御，不是已确认的 live leak**（据实表述，不过度宣称）。

- **console-api memstore 缓存是 FIFO 而非 access-order LRU（真实，fallback-mode-only）**：`internal/consoleapi/memstore.go` 的 `cacheChunkUnlocked`（:76-91）/ `cacheTraceUnlocked`（:96-111）是 FIFO（既有 key 覆盖 **不** reorder——:80-83/:100-103 早返回；驱逐弹 `[0]`，:87/:107）；读路径 `GetSourceChunk`（:341-352）/ `GetSearchTrace`（:357-368）**不**做 move-to-front。`types.go:69` / `handlers.go:277` 注释已写 "in-memory LRU"（术语已超前）。cap = `resolveCacheCapacity`（Phase 31，env `CONTEXTFORGE_CONSOLEAPI_CACHE_CAP`，默认 256）。**fallback-mode only**（生产接真实 backend）。

- **memory hard-delete cascade（HONEST 非问题——无可级联）**：`core/src/memory/store.rs` `hard_delete`（:235-246）做单条 `DELETE FROM memory_items`。**全 schema 审计**（6 表 / migration 0010-0018）：`memory_id` 仅在 `memory_items` 是 PK；**无其它表带 memory_id**；**无 memory-vector / embedding 表**（grep=0）；vector-store 层与 memory 零耦合。故无任何孤儿 → 写 cascade 代码 = 推测性 / 不可能场景（CLAUDE.md Simplicity-First）。

- **handleMemoryPin strict-400（DROPPED——刻意契约，不改）**：`handlers.go:521-549`：malformed / empty / absent body 刻意回落 `pin=true` 以保 v0.7 lenient contract（doc-comment :519-524，task-17.1 / ADR-022 D2）。改 400 会违 ADR-004 + 与已 Accepted 的 ADR-022 D2 矛盾。记为 HONEST non-change（无代码改）。

- **indexing.* 事件无持久化 / 无 replay（REAL gap）**：indexing 事件（indexing.progress / .cancelled / .error）仅广播到内存 `EventBus`（`core/src/jobs/index_session_backend.rs:157-218` `build_*_event`→`eb.send`，best-effort，不持久化）；`replay_events_from_audit`（`core/src/data_plane/events.rs:391-420`）+ `audit_op_str_to_event`（:369-377）**仅处理 memory_***（marker `[SPEC-DEFER:phase-future.indexing-event-persistence]` @ events.rs:389）。`AuditOperation` enum（`core/src/memoryops/audit.rs:12-20`）**无 indexing variant** + `AuditLogEntry` 缺 `job_id/processed/total`。

- **TraceStore 无多 workspace 隔离（add-only proto + SQL filter）**：`core/src/data_plane/search_persist.rs` `get`（:129-142）/ `list`（:147-174）/ `search_fts`（:219-259）/ `load_warm`（:184-213）**无 `WHERE workspace_id`**；内存 TraceStore `search.rs` `get/list` 亦未过滤；handlers `get_search_trace`（:460-480）/ `list_queries`（:486-502）忽略 workspace。`GetSearchTraceRequest`（proto:237-239）+ `ListQueriesRequest`（:255-257）**缺 workspace_id field**（marker task-16.1:288）。

- **events-drain-timeout-config（VERIFY-ONLY——Phase 26 已交付）**：`CONSOLE_EVENTS_DRAIN_TIMEOUT` 真实存在（`internal/consoleapi/grpcclient/grpcclient.go:405-419` `drainTimeoutFromEnv`，默认 100ms）+ 通过测试 `TestDrainTimeoutFromEnv`（grpcclient_test.go:867-895）。本 phase 据实从 add 改 verify（与 Phase 31 event-bus-partition verify-only 校正同形），不重实现，引用既有测试。

- **export --timeout（GENUINE nit）**：`internal/cli/export.go:29` 硬编码 `context.WithTimeout(..., 60*time.Second)`；`parseExportOpts`（:58-91）无 `--timeout`。task-31.3 后 export 做**两次顺序 daemon spawn**（search.go source.go:91 spawn#1 + listChunks source.go:104-109 spawn#2）于一个 60s ceiling 下（各最多等 `daemonHealthDeadline=15s`）→ 可能偏紧。

本 ADR 把上述「L2 缓存无界 / memstore FIFO→LRU + hard-delete invariant / indexing 持久化 + replay / TraceStore 隔离 / drain verify-only / export timeout」收敛为一个集中清理 + 诚实化 Phase 的处理策略。改动**多为 code-local 🟢 可单测**；indexing replay e2e（restart）/ TraceStore 多 workspace e2e console 须 running daemon → 🟡 honest-defer，不伪造（ADR-013）。全部改动遵守 ADR-004 默认行为 / proto / migration（add-only）/ 既有契约不变 + 0 网络 + ADR-008 0 新依赖 + ADR-013 受阻 / 非问题 / 已交付项诚实分级不伪造。

## Decision

第二轮治理债清理采用 **「缓存有界化 + access-order LRU + observability 持久化与隔离 + 契约诚实化 + 默认零依赖守线」** 策略，分 5 个决策点：

### D1 — L2 embedding cache rowid-FIFO bound（opt-in-path caveat；true-LRU honest-defer）（task-33.1）🟢

为 L2 SQLite embedding 缓存加行数上界 + rowid-FIFO 驱逐：在 `sqlite_put`（cache.rs:153-161）写后 `COUNT(*)`；超 cap 则 `DELETE ... WHERE rowid IN (SELECT rowid ... ORDER BY rowid ASC LIMIT overflow)`（用隐式 rowid 作 FIFO 序）。L2 cap 默认 const 与 `DEFAULT_EMBEDDING_CACHE_CAP`（:23）并列。**0 新依赖、0 schema migration**（用隐式 rowid；带 `created_at` 列的真 LRU 须对既有用户文件 `ALTER` → honest-defer `[SPEC-DEFER:phase-future.l2-cache-true-lru]`）。public ctor（`new` :89 / `with_sqlite` :105 / `with_capacity` :95）保持 source-compatible。镜像 TEST-31.2.1（cache.rs:345）。

**理由**：L1 已是 BoundedCache FIFO（Phase 31 / ADR-027），但 L2 `INSERT OR REPLACE`（:155）无 cap → 行只增不减，是治理对称缺口——L1 有界 / L2 无界。用隐式 rowid 作 FIFO 序的 row-count cap + 驱逐最 surgical：0 新 dep、0 schema migration（不 `ALTER` 既有用户文件）。**诚实 caveat（ADR-013）**：`with_sqlite` 无生产调用点（test-only，:331/337），shipped daemon 用 memory-only L1 → 这是 **opt-in-path 纵深防御，不是已确认 live leak**，spec / ADR 据实表述、不夸大为生产泄漏。带 `created_at` 真 LRU 须 `ALTER` 既有文件（破 0-migration） → honest-defer `[SPEC-DEFER:phase-future.l2-cache-true-lru]`。备选「不 bound L2，依赖 L1」忽略 opt-in 路径的纵深防御价值，不取。

### D2 — memstore access-order LRU + hard-delete no-dangling-ref invariant（cascade honest-defer 非问题；handleMemoryPin lenient ADR-022 D2 保持）（task-33.2）🟢

(a) **memstore FIFO → access-order LRU**：`memstore.go` `cacheChunkUnlocked`（:76-91）/ `cacheTraceUnlocked`（:96-111）升级为 access-order LRU——命中时 move-to-front（读路径 `GetSourceChunk` :341-352 / `GetSearchTrace` :357-368）+ 既有 key 覆盖时 move-to-front。cap = `resolveCacheCapacity`（Phase 31，env `CONTEXTFORGE_CONSOLEAPI_CACHE_CAP`，默认 256）。**须改既有 `TestMemStore_CacheEviction_FIFO`**（memstore_test.go:209-243，硬断言 FIFO）→ LRU。0-dep，fallback-mode only（生产接真实 backend）。
(b) **memory hard-delete cascade = HONEST-DEFER（非问题，无可级联）**：`store.rs` `hard_delete`（:235-246）单条 `DELETE FROM memory_items`。全 schema 审计（6 表 migration 0010-0018）：`memory_id` 仅 `memory_items` PK，无其它表带 `memory_id`，无 memory-vector/embedding 表（grep=0），vector-store 层零 memory 耦合 → 无孤儿 → 写 cascade = 推测 / 不可能场景（CLAUDE.md Simplicity-First）。交付物 = 一个 **INVARIANT 测试**（schema introspection：`memory_items` 是唯一带 `memory_id` 的表 + `get(id)=None` after hard_delete），使未来若加 `memory_id` FK 会令其失败而强制真实决策。tag `[SPEC-DEFER:phase-future.memory-harddelete-cascade]`（only-if-future-FK）。
(c) **handleMemoryPin strict-400 = DROPPED（刻意契约，不改）**：`handlers.go:521-549` malformed / empty / absent body 刻意回落 `pin=true` 保 v0.7 lenient contract（doc-comment :519-524，task-17.1 / ADR-022 D2）。改 400 违 ADR-004 + 与已 Accepted ADR-022 D2 矛盾 → 记 HONEST non-change（无代码改），D4 重申其按设计保持 lenient。

**理由**：(a) memstore 注释已写 "in-memory LRU"（types.go:69 / handlers.go:277）但行为是 FIFO（既有 key 覆盖不 reorder、读不 move-to-front）——术语与行为不一致是治理债，move-to-front 修齐，0-dep，fallback-mode only。须同步改硬断言 FIFO 的旧测（memstore_test.go:209-243）→ LRU。(b) hard-delete cascade 经全 schema 审计确认**无可级联**（无第二张表带 memory_id、无 memory 向量表）——写 cascade 代码是不可能场景的防御，违 Simplicity-First；据实交付 invariant 测试（守 "memory_items 是唯一带 memory_id 的表" + hard_delete 后 get=None），把真实工作（仅在未来加 FK 时才需）honest-defer `[SPEC-DEFER:phase-future.memory-harddelete-cascade]`。(c) handleMemoryPin lenient 是 ADR-022 D2 刻意契约（task-17.1 doc-comment 明记），改 400 会破已 Accepted 决策 + ADR-004 既有契约——据实记为 non-change，不动代码。

### D3 — observability：indexing event 持久化 + replay（add-only migration 0019）+ TraceStore workspace 隔离（add-only proto field）+ drain-timeout verify-only 校正（task-33.3）🟢 / 🟡

(a) **indexing event 持久化 + replay**（REAL gap）：indexing 事件（indexing.progress / .cancelled / .error）当前仅广播至内存 `EventBus`（`index_session_backend.rs:157-218` `build_*_event`→`eb.send`，不持久化）；replay mapper（events.rs:391-420 + :369-377）仅 memory_*（marker @ events.rs:389）。Fix：**add-only migration 0019_indexing_events**（专用表，比复用 audit 更干净——推荐），在 emit site 持久写入 indexing job 生命周期（`job_id/stage/processed/total/ts`）+ 扩 replay mapper 以 id/ts ASC 重建 indexing.* PbEvents。mapper 单测 🟢（纯函数，镜像 TEST-26.2.3 events.rs:493）；端到端 restart 🟡（须 running daemon / job runner `[SPEC-DEFER:phase-future.indexing-replay-e2e]`）。**真实 job_id/processed/total 取自持久行，不合成**（ADR-013）。
(b) **TraceStore 多 workspace 严格隔离**（add-only proto + SQL filter）：`search_persist.rs` `get`（:129-142）/ `list`（:147-174）/ `search_fts`（:219-259）/ `load_warm`（:184-213）+ 内存 `search.rs` get/list 无 `workspace_id` 过滤；handlers `get_search_trace`（:460-480）/ `list_queries`（:486-502）忽略 workspace。Fix：**add-only `workspace_id` field on `GetSearchTraceRequest`（proto:237-239）+ `ListQueriesRequest`（proto:255-257）**（buf generate proto 重生）+ 接线到 TraceStore / SqliteTracePersist + 加 `WHERE workspace_id` 过滤；**EMPTY workspace_id 须保持当前 aggregate-all 行为**（ADR-004 back-compat）。marker `[SPEC-DEFER:phase-future.tracestore-multi-workspace-strict]`（task-16.1:288）。SQL / handler 测 🟢；e2e console 🟡。
(c) **events-drain-timeout-config = VERIFY-ONLY**（Phase 26 已交付）：`CONSOLE_EVENTS_DRAIN_TIMEOUT` 真实（grpcclient.go:405-419 `drainTimeoutFromEnv`，默认 100ms）+ 通过 `TestDrainTimeoutFromEnv`（grpcclient_test.go:867-895）。据实从 add 改 verify（与 Phase 31 event-bus-partition verify-only 校正同形），**不重实现，引用既有测试**。

**理由**：(a) indexing.* 仅广播不持久化 + replay mapper 仅 memory_* 是真实 observability gap（marker @ events.rs:389 已标）；专用 add-only migration 0019 比复用 `AuditOperation` enum + `AuditLogEntry` 更干净（后者缺 job_id/processed/total），在 emit site 持久写 + 扩 mapper，真实数值取持久行不合成（ADR-013）；e2e restart 须 running daemon → honest-defer `[SPEC-DEFER:phase-future.indexing-replay-e2e]`。(b) TraceStore 全读路径无 workspace 过滤 + 两 request message 缺 workspace_id 是真实隔离 gap（marker task-16.1:288）；add-only proto field（不改既有 tag）+ SQL `WHERE` 过滤最 surgical，且 EMPTY workspace_id 保 aggregate-all（ADR-004 back-compat、不破既有 client）；e2e console 须 running daemon → 🟡。(c) drain-timeout 经核 Phase 26 已交付 + 有通过测试——据 ADR-013 据实从 add 改 verify-only（镜像 Phase 31 event-bus-partition 校正），引用既有测试，不重实现。

### D4 — honest-defer 边界 + dropped-nits 诚实（task-33.1 / 33.2 / 33.3 / 33.4）🟡 / non-issue / non-bug

诸受阻 / 非问题 / 已交付 / 刻意契约项据 ADR-013 如实记录、不伪造完成、不夸大：
- **L2 cache true-LRU**（带 created_at 列）须 `ALTER` 既有用户文件 → honest-defer `[SPEC-DEFER:phase-future.l2-cache-true-lru]`（D1 用隐式 rowid FIFO，0-migration）。
- **L2 cache opt-in-path caveat**：`with_sqlite` 无生产调用点（test-only）→ 据实记为纵深防御，不夸大为 live leak（ADR-013）。
- **memory hard-delete cascade = 非问题**（全 schema 审计无可级联）→ honest-defer `[SPEC-DEFER:phase-future.memory-harddelete-cascade]`（only-if-future-FK），交付 invariant 测试而非 cascade 代码。
- **handleMemoryPin lenient = 刻意契约（ADR-022 D2）保持**——dropped，不改代码（改 400 会违 ADR-004 + ADR-022 D2）。
- **indexing replay e2e（restart）** 须 running daemon / job runner → 🟡 honest-defer `[SPEC-DEFER:phase-future.indexing-replay-e2e]`（mapper 🟢 已达 / e2e 未跑、不预填）。
- **TraceStore 多 workspace e2e console** 须 running daemon → 🟡 honest-defer `[SPEC-DEFER:phase-future.tracestore-multi-workspace-strict]`（SQL/handler 🟢 已达 / e2e 未跑、不预填）。
- **events-drain-timeout = VERIFY-ONLY**（Phase 26 已交付，引用既有测试，不重实现）。
- **dropped-nits 诚实（grounding 校正，记于 §范围外，不实现）**：(i) `%v→%w` in `internal/daemon/search.go` 是 **NON-BUG**（该文件无 fmt；真正的 `%v` 在 `internal/cli/search.go:88`——一个 terminal `Fprintf`，`%w` 在此 **INVALID/vet-error**，且 `err.Error()` 已携完整 grpc Status → grpc Status **未丢**）；(ii) tracestore-fts 跨版本 migration **已修 + 已测**（`search_persist.rs:84-90` + `backfill_fts_if_empty` :304，TEST-26.1.4/.4b）→ no-op；(iii) datadir env-global → `daemon.Options.DataDir` 是 **REAL 但 🟡**（`cmd/contextforge/main.go:254` `setDataDirEnv` 是刻意的 Go→Rust 跨进程 datadir 传输经 `CONTEXTFORGE_DATA_DIR`；`daemon.Options.DataDir` field 须接入 child `cmd.Env` + 改 spawn 契约）→ honest-defer `[SPEC-DEFER:phase-future.daemon-options-datadir]`。

**理由**：据 ADR-013，对受阻 / 非问题 / 已交付 / 刻意契约项诚实记录边界、不伪造完成、不夸大缺口。这是本轮治理债清扫的核心价值——grounding 校正掉 survey 高估：cascade 是非问题（无可级联）、handleMemoryPin 是刻意契约、drain-timeout 已交付、`%v→%w` 是 non-bug（该文件无 fmt + grpc Status 未丢）、tracestore-fts 已修已测、datadir 是 REAL 但 🟡——逐项据实分级而非一律「修」，符合 Simplicity-First + ADR-013。真实工作（true-LRU / e2e / datadir Options）据实 defer 并打 SPEC-DEFER tag，不预填、不沿用误导表述。

### D5 — 默认行为 + proto + migration（add-only）+ 0-dep / 0-network 不变（all tasks）🟢

所有改动保持默认行为 / proto / migration（add-only）/ 既有契约不变 + 0 网络（ADR-004）+ 0 新依赖（ADR-008，Phase 33 = 0 dep）：L2 cache bound 默认 const + rowid-FIFO（0 新 dep、0 schema migration，public ctor source-compatible）；memstore access-order LRU 是 fallback-mode 内部行为升级（0-dep，生产接真实 backend）；hard-delete invariant 是新增测试（无行为改）；migration 0019_indexing_events 为 add-only 专用表（不改既有表）；TraceStore `workspace_id` 为 add-only proto field（不改既有 tag）+ EMPTY workspace_id 保 aggregate-all（既有 client 不破）；drain-timeout verify-only（无代码改）；export `--timeout` 为 add-only flag（默认 60s，default byte-equiv）。既有 `cargo-test` / `go-test` / `spec-lint` 三门不退化。

**理由**：ADR-004 local-first + ADR-008 dep add-only——默认行为 / proto / migration（add-only）/ 既有契约不变 + 0 网络 + 0 新依赖是不可让渡 baseline。本 phase 为治理债清理 + 诚实化——非默认行为演进。rowid-FIFO（0-migration）/ access-order LRU（fallback-mode 内部）/ add-only migration 0019（专用表）/ add-only proto field（EMPTY 保 aggregate-all）/ add-only `--timeout`（default byte-equiv）使既有用户与既有契约零感知。

## Consequences

## Alternatives

- **A1（不 bound L2 cache，依赖 L1）**：保留 L2 `INSERT OR REPLACE` 无 cap，依赖 L1 BoundedCache 兜底。否决：L1 与 L2 是独立层，L2（opt-in SQLite 文件）行只增不减是真实治理对称缺口；rowid-FIFO row-count cap 0 新 dep / 0 migration 即修，且 opt-in-path 的纵深防御有价值（据实标注非 live leak，不夸大）。
- **A2（带 created_at 列的真 LRU）**：给 `embedding_cache` 加 `created_at` 列 + 真 LRU 驱逐。否决：须对既有用户 SQLite 文件 `ALTER`（破 D1 的 0-migration），跨既有文件 schema 演进超本治理项范围；据 D1 用隐式 rowid 作 FIFO 序（0-migration），真 LRU honest-defer `[SPEC-DEFER:phase-future.l2-cache-true-lru]`。
- **A3（为 memory hard-delete 写 cascade 代码）**：直接写跨表 cascade 删除。否决：全 schema 审计（6 表 migration 0010-0018）确认无第二张表带 memory_id、无 memory 向量表、vector-store 零 memory 耦合 → 无孤儿，cascade 代码是不可能场景的防御（违 CLAUDE.md Simplicity-First）；据 D2 交付 invariant 测试（守唯一性 + get=None），真实 cascade 仅在未来加 FK 时才需 → honest-defer。
- **A4（handleMemoryPin 改 strict-400）**：malformed / empty body 返 400。否决：v0.7 lenient contract 是 ADR-022 D2 刻意决策（task-17.1 doc-comment :519-524），改 400 会破已 Accepted 决策 + ADR-004 既有契约——据 D2/D4 记为 HONEST non-change，不动代码。
- **A5（indexing replay 复用 audit_log 而非专用 migration 0019）**：扩 `AuditOperation` enum + 在 audit_log 记 indexing 事件。否决：`AuditLogEntry` 缺 job_id/processed/total（须改既有 audit schema），且 indexing 生命周期语义与 memory state-op audit 异质；据 D3 用 add-only 专用表 migration 0019_indexing_events 更干净（不改既有 audit 表 / enum）。
- **A6（TraceStore workspace 隔离用 hard-filter，EMPTY 也过滤）**：EMPTY workspace_id 时返回空集。否决：会破既有 client（既有调用不传 workspace_id）+ 违 ADR-004 back-compat；据 D3 EMPTY workspace_id 保 aggregate-all（既有行为不变），仅在显式传 workspace_id 时严格过滤，add-only field 不破既有 client。
- **A7（重实现 events-drain-timeout）**：把 drain-timeout 当新功能再实现一遍。否决：经核 Phase 26 已交付（grpcclient.go:405-419 + TestDrainTimeoutFromEnv :867-895）；据 ADR-013 从 add 改 verify-only（镜像 Phase 31 event-bus-partition 校正），引用既有测试，重实现 = 冗余 + 违 Surgical-Changes。
- **A8（按 survey 实现 `%v→%w` / tracestore-fts migration / datadir Options）**：把三 dropped-nits 都「修」。否决：grounding 校正——`%v→%w`（daemon/search.go）是 non-bug（该文件无 fmt；真正 `%v` 在 cli/search.go:88 terminal Fprintf，`%w` 在此 INVALID/vet-error 且 grpc Status 已经 `err.Error()` 携带未丢）/ tracestore-fts 跨版本 migration 已修已测（search_persist.rs:84-90 + :304，TEST-26.1.4/.4b）no-op / datadir env→Options 是 REAL 但 🟡（须接 child cmd.Env + 改 spawn 契约）→ honest-defer。据 ADR-013 逐项据实分级而非一律实现。

## Amendment (Phase 40 / v0.33.0) — l2-cache-true-lru 兑现 + 真-LRU 假设据实更正 (add-only)

> add-only Amendment（不溯改本 ADR D-body / §A2 / §D4 / Ratification (v0.26.0)，ADR-014 D5）。承本 ADR §D1（L2 rowid-FIFO bound）+ §A2/§D4 据「带 created_at 列的真 LRU 须 ALTER 既有用户文件 → 破 0-migration」延后的 `[SPEC-DEFER:phase-future.l2-cache-true-lru]`。

Phase 40 / v0.33.0（ADR-045 D2）兑现 L2 缓存访问序 LRU 维度并据实**更正本 ADR §A2/§D4 的延后假设**：本 ADR 当时判定真-LRU 须新增 `created_at` 列 + ALTER 既有文件——grounding 更正：访问序 LRU **不**须时间列。task-40.2（PR #258）在 `sqlite_get` 命中分支（仅 `l2_cap > 0`）对命中行 `INSERT OR REPLACE` 原样回写（`INSERT OR REPLACE` 对既有 PK 行先 DELETE 再 INSERT → 获新最大 rowid），把隐式 rowid 由插入序变访问序，既有 `sqlite_put` 的 rowid 序驱逐（本 ADR §D1）随之由插入序 FIFO 升访问序 LRU。**复用既有隐式 rowid、0 schema migration**，与 Go memstore 命中 move-to-front（本 ADR §D2 / task-33.2）同技法。

本 ADR §D1 的 rowid-FIFO（row-count cap 本身）仍是正确且必要的前序；Phase 40 仅补「命中 bump 使 rowid 序由插入序变访问序」增量。**诚实校正（ADR-013）**：命中 bump 给 L2 读路径加一次行重写（写放大），是访问序 LRU 固有代价；`with_sqlite` 无生产调用点（本 ADR §D1 已据实标注 opt-in-path）→ 现网零影响，本项是 opt-in 路径语义补全、非已确认线上问题。cap==0（不限）不 bump（保插入序、零额外写）。验证 TEST-40.2.1/40.2.2。详见 ADR-045 Ratification (v0.33.0) + `docs/releases/v0.33.0-evidence.md`。
