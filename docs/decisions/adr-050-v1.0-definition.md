# ADR `050`: `v1.0-definition`

**Status**: Proposed（Phase 45 / v0.38.0：D1 能力 + D2 API/CLI 冻结已交付 → 部分 ratify；Phase 46 / v0.39.0：D3 文档 + D4 发布流程已交付 → 部分 ratify 扩展至 D1/D2/D3/D4；v1.0.0 完整 ratify Proposed→Accepted 待 Phase 47，见 §Ratification）

**Category**: 项目里程碑定义 / 版本策略 / API 冻结 / 收口
**Date**: 2026-07-01
**Decided By**: 主 agent（ADR-012 自治，用户授权 v1.0 务实收口）
**Related**: ADR-007（minimal-tarball-distribution — 把 v1.0 列为"多平台+签名+自动更新+企业部署"分发目标，本 ADR 收窄为务实收口，自动更新/multi-user 推 v2.0）/ ADR-013（禁伪造红线 — v1.0 锚点必须真实可验证，honest-defer 项显式列已知限制）/ ADR-015（console-contract-v1 proto FROZEN — API 冻结的 proto 面已就绪）/ ADR-017（console-contract-completion — 悬空的 "v1.0 release gate" 提法，本 ADR 正式承接为可执行锚点）/ ADR-004（local-first-privacy-baseline — v1.0 守 0-network/0-dep 默认 baseline）/ ADR-014（D1-D5，第三十六次激活）/ roadmap §v1.0 锚点段

## Context

ContextForge 截至 Phase 44（memory-unpin-actor-propagation, Done / v0.37.0）。**项目从未立过 v1.0 锚点**——四处查证一致（ADR-013 grounding）：

- **PRD** 的 P0 清单是 v0.1 的（早已全部满足且远超）；PRD 里 "v1.0" 只出现在**分发维度**（`§Constraints 发布`："v1.0 多平台 release + 签名校验 + 自动更新 + 企业部署文档"），从未把 v1.0 当作能力里程碑定义。
- **roadmap** 全篇零 "v1.0" 提及；开放式外推（Phase N → v0.(N-7).0）无终止条件。
- **README** 无成熟度标签（无 alpha/beta/stable），状态段冻结在 "v0.2 limitations"。
- **ADR-017** 出现过一句悬空的 "v1.0 release gate"（"留 ADR-018+ retrospective"），但从未被一份 ADR 正式承接。

同时，项目实际能力**远超 v0.1 P0**：语义检索（hybrid RRF + reranker + 远程 embedding/reranker live）、向量后端（qdrant/lancedb/sqlite-vec/hnsw）、Console Contract v1（26 endpoint）、memory lifecycle（pin/unpin/deprecate/soft/hard-delete + audit/event source 归因）、code_cjk tokenizer 默认开启——这些 PRD 里是 v0.1 P1 stretch 或"留 v0.2+"的，现在都做了。实测 recall@5/@10=1.0（超 PRD Success Metrics 北极星 Top-5≥75% / Top-10≥85%）。

**问题**：没有 v1.0 定义 = 没有 checklist 可对照打勾 = "冲刺 v1.0" 无判定标准。本 ADR 正式立锚点。

## Decision

**v1.0.0 = 功能成熟度收口 + API/CLI 冻结 + 文档对齐 + GitHub Release 流程**。分 4 个锚点维度 + 明确不含清单：

### D1 — 能力锚点（已满足，v1.0 不新增核心能力）

v1.0 的能力门槛是 **v0.1 P0 全部满足**（Phase 1-8 早已闭环）+ PRD 核心能力 5 条全交付。当前 v0.37.0 **远超**此门槛（语义/hybrid/reranker/向量/console 全交付，recall@5/@10=1.0）。**v1.0 不新增核心能力**——它是收口里程碑，不是功能里程碑。能力侧的 SPEC-DEFER 项（大语料质量基准 / hnsw 持久化 / vector-dim-enforce 等）不阻塞 v1.0，列为"已知限制"。

**理由**：PRD 核心能力 ≤5 条已收敛；recall 实测超北极星；继续加功能是 v1.x/v2.0 的事，v1.0 是"锁定现有成熟度"。

### D2 — API/CLI 冻结锚点（Phase 45 交付）

v1.0 意味着 **API/CLI 稳定承诺**（v1.x 不 breaking，breaking 推 v2.0）。冻结前清理：
- **proto 契约**：已 FROZEN（ADR-015，7 proto 全 FROZEN 声明 + add-only 演进纪律最强）——v1.0 就绪。
- **daemon REST**：移除 2 个 501 未实装 端点（`POST /v1/import` + `POST /v1/eval/run`，§2A 决策 B 有意留下，console-api 已完整覆盖 import/eval）；实装 `chunk_count`（真实 COUNT 查询，非 placeholder 0）——Phase 45 task-45.2。
- **CLI**：加 `--version`/`version` 子命令（v1.0 产品必须有版本可查）+ 顶层 `--help`（修复 `-h` 落 unknown subcommand exit 2）；example.toml 补全 4 个检索 section（`[embedding]`/`[vector]`/`[reranker]`/`[retrieval]`）——Phase 45 task-45.3。

**理由**：API 冻结一个永久 501 端点不可接受（承诺永不实装的接口）；v1.0 产品无 `--version` 是硬伤。daemon REST 移除是 v1.0 前允许的 breaking change（major version 边界）。

### D3 — 文档锚点（Phase 46 交付）

v1.0 意味着 **新手能凭文档上手**。文档对齐：
- README 重构：删 85% 变更日志（迁 RELEASE_NOTES）+ 加 `## Features` 汇总 + quickstart 补 export + 刷新所有 v0.28.0→v1.0 + 移除 "v0.2 limitations" 段——Phase 46 task-46.1。
- production.md / compose 刷新版本 pin + ADR 索引 + CHANGELOG.md——Phase 46 task-46.2。

**理由**：README 落后 9 版本（v0.28.0）+ 无 Features 汇总，新手无法评估能力。v1.0 必须文档与实际一致。

### D4 — 发布锚点（Phase 46/47 交付）

v1.0 意味着 **发布产物完整可信**。发布流程：
- release.yml 加 GitHub Release 对象自动创建（tarball + 签名）——当前只有 GHCR 镜像，无 GitHub Release 页面——Phase 46 task-46.3。
- v1.0.0 tag（major version bump）+ 完整 GitHub Release——Phase 47 task-47.2。
- 供应链安全（cosign + SBOM + provenance）每次 release 自动跑——已就绪（v0.21.0 起）。

### 不含清单（honest-defer 推 v2.0，ADR-013）

v1.0 **不含**以下（在 v1.0 release notes 显式列为"已知限制/未来路线"）：
- **multi-user / 认证身份 / 权限 / 审计合规**——PRD §Out of Scope 明确 v0.1 不做；ADR-016/018 反复"留 v1.0"但工程量大（Postgres/权限层/审计），务实收口推 v2.0。
- **自动更新机制**——PRD §Constraints 发布列 v1.0 目标，但无 task/ADR 兑现，从零工程，推 v2.0。
- **arm64 native runner / 多平台 native 构建**——ADR-033 实测 QEMU 不可行，需原生 runner，推 v2.0。
- **大语料向量质量基准 / hnsw 图持久化 / vector-dim-feature-enforce / SSE live e2e** 等能力侧 SPEC-DEFER——不阻塞 v1.0，列为已知限制。

**理由**：把这些塞进 v1.0 会让收口无限延期（roadmap §3.17/§3.22 "据实排小不凑数"）。v1.0 是"锁定现有成熟度的可信发布"，multi-user 等是 v2.0 的产品演进。

## Consequences

- **Positive**: v1.0 锚点正式确立——后续 Phase 45/46/47 有可对照的 checklist（D1-D4）；v0.1 P0 能力早已超满足，v1.0 是收口而非功能冲刺；API/CLI 冻结给用户稳定承诺；文档对齐让新手能上手；GitHub Release 流程让发布可信。
- **Negative / open**: v1.0 不含 multi-user / 自动更新 / arm64 native——这些 honest-defer 推 v2.0（ADR-013 显式记录，不伪造）；现有 SPEC-DEFER 项列为"已知限制"非"已解决"。
- **Ratification**: 本 ADR **Proposed**。Phase 45（D2 API/CLI 冻结）+ Phase 46（D3 文档 + D4 发布流程）交付后，于 Phase 47 / v1.0.0 正式发版完整 ratify Proposed→Accepted（逐 D 据真实交付验证）。
- **Follow-ups**: v2.0 路线（multi-user / 自动更新 / arm64 native / 大语料质量）在本 ADR §不含清单记录，由 v1.0 后的 ADR-051+ 承接。

## Alternatives

- **A1（v1.0 含 multi-user）**：把 ADR-016/018 反复"留 v1.0"的 multi-user 纳入。否决：multi-user 是中型 feature（Postgres/权限/审计合规），工程量大且 PRD §Out of Scope 明确 v0.1 不做；塞进 v1.0 会让收口无限延期。务实收口推 v2.0。
- **A2（v1.0 = PRD §Constraints 分发定义：多平台+签名+自动更新+企业部署）**：严格按 PRD 原始定义。否决：arm64 native 受阻（ADR-033）+ 自动更新从零，分发维度收口成本高且非能力门槛。本 ADR 收窄 v1.0 为"务实收口"，自动更新/multi-platform native 推 v2.0。
- **A3（不立 v1.0，继续 v0.x 无限外推）**：保持 roadmap 开放式外推。否决：项目已到 v0.37（远超 PRD Phase 17/v0.10 边界），无 v1.0 锚点 = 无收口判定 = 无稳定承诺。v1.0 是给用户/自己的成熟度信号。
